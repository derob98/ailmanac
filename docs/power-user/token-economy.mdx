You pay for every token in and every token out. The good news: most real workloads are carrying **dead weight** — bloated system prompts, re-sent context, verbose replies, the wrong model for an easy job. Trim that and the bill drops without touching quality. This page is the power-user toolkit, ordered roughly by leverage.

## First, find where the tokens go

Before optimizing, split your spend into three buckets — each has a different fix:

<Flashcards title="The three token buckets" cards={[{front:"Input tokens",back:"Everything you send: system prompt, tools, history, documents, the user turn. Often the biggest and most ignored cost in long sessions."},{front:"Output tokens",back:"Everything Claude generates. Priced higher than input. The target of terse-style prompting."},{front:"Reused context",back:"The stable chunk you re-send every call (system prompt, tool catalog, big doc). Pure waste at full price unless you cache it."}]} />

The fixes line up cleanly: **cache** the reused context, **trim** the input, **shorten** the output, **right-size** the model, and **batch** what isn't time-sensitive.

## The terse / "caveman" style (output savings)

The viral move is telling Claude to drop filler and answer in fragments — popularized by the open-source [`caveman`](https://github.com/JuliusBrussee/caveman) Claude Code skill (MIT-licensed, by Julius Brussee), whose tagline is "why use many token when few token do trick." It forces short sentences, infinitive verbs, and zero pleasantries.

<VerifyNote lastVerified="2026-06-24" source="https://github.com/JuliusBrussee/caveman">
The `caveman` skill reports ~65% average output-token reduction across its 10-task benchmark (range 22–87%) and notes it only shrinks **output** tokens — thinking and input are untouched. Independent benchmarks on already-concise structured tasks found far smaller gains (roughly 9–21%). Treat the headline percentage as workload-dependent, not a guarantee.
</VerifyNote>

The honest takeaway from independent testing: the style **costs nothing on content** (code, technical terms, JSON stay exact) but the savings depend entirely on your baseline. If your prompts already say "be concise," most of the win is already banked. The big reductions (40–65%) show up on explanation-heavy answers; structured extraction barely moves.

<PromptCard title="Lean instruction block — paste into your system prompt">
{`Answer terse. Cut filler, hedging, and pleasantries.
Drop articles (a/an/the) and softeners (just, really, basically, actually).
No preamble, no restating the question, no "happy to help."
Fragments are fine. Keep technical terms and code blocks exact.
Pattern per point: [thing] [action] [reason]. Next step if any.`}
</PromptCard>

<Callout type="tip" items={["Put the terse rule once in the system prompt, not in every user turn — repeating it re-pays the input cost each call.","Never compress code, identifiers, JSON, or numbers. Compress prose only.","'Be concise. Return JSON only.' is itself ~60% of the achievable output savings — write it before reaching for fancier tricks."]} />

<Callout type="warning" items={["Terse style trims OUTPUT only. It does nothing for a 20k-token system prompt you re-send every call — that's an input/caching problem.","On extended-thinking tasks, the reasoning tokens are unaffected; you only shrink the final visible answer."]} />

## Compress tool output before it re-enters context (RTK)

Caveman shrinks what Claude *writes*. The mirror-image leak in agentic sessions is what Claude *reads back*: every `git status`, `npm test`, `ls -la`, or build log a coding agent runs gets piped — headers, progress bars, and all — straight into the context window as input tokens on the next turn. On a long [Claude Code](/docs/claude-code/what-is-claude-code) session that observation noise often dwarfs the system prompt.

[`rtk`](https://github.com/rtk-ai/rtk) ("Rust Token Killer", Apache-2.0, in homebrew-core) is a CLI proxy aimed at exactly this. `rtk init -g` installs a Claude Code **[PreToolUse hook](/docs/claude-code/hooks) on the Bash tool** (`rtk hook claude`) that transparently rewrites each command to `rtk <command>` and compresses its output — stripping headers and visual noise, grouping similar lines, deduplicating repeats with counters, and truncating redundancy — before it ever reaches the model.

<VerifyNote lastVerified="2026-06-24" source="https://github.com/rtk-ai/rtk">
RTK's project claims 60–90% output reduction on common dev commands. That figure is the tool's own benchmark, not independently verified, and it only touches the **Bash** tool — Claude Code's native Read, Grep, and Glob bypass the hook (route them through it with the shell equivalents `cat`/`rg`/`find`, or `rtk read`/`rtk grep`/`rtk find`). At least one reported issue (#582) found the hook *raising* total cost ~18% in some setups, so measure on your own workload before trusting the headline.
</VerifyNote>

<PromptCard title="Install RTK and wire the global hook (macOS)">
{`brew install rtk          # see the repo for cargo / other installers
rtk init -g --hook-only   # PreToolUse(Bash) hook only, no RTK.md file
# undo anytime: rtk init -g --uninstall`}
</PromptCard>

The `--hook-only` flag matters for the token budget: the default `rtk init` also writes an `RTK.md` instruction file that loads every session — a fixed *input* cost that partly fights the savings. Hook-only keeps the output compression and adds nothing to the prefix. Restart Claude Code after `rtk init -g` for the hook to load.

<Callout type="tip" items={["RTK and caveman are complementary, not redundant: caveman compresses model OUTPUT, RTK compresses tool-result INPUT (the observations fed back each turn). Stacking them covers both directions.","The hook fires per Bash call and rewrites transparently — you do not change how you prompt.","Measure with the real bill, not the README. If your sessions are light on shell commands, RTK has little to compress — its win scales with how noisy your tool output is."]} />

## Cache the reused prefix (input savings)

If many calls share a large unchanging chunk — a long system prompt, a tool catalog, a reference document — **prompt caching** processes it once and reuses it at a fraction of the input price on every later call. This is the single highest-leverage structural change for chat and agent workloads, because it pays back on every turn.

The one rule: the cached prefix must be **byte-for-byte identical** across calls. A stray timestamp or reordered tool list near the top silently drops your hit rate to zero. Full mechanics, the copy-paste `cache_control` snippet, and how to verify hits live in [Prompt Caching & Cost Optimization](/docs/api/prompt-caching).

## Trim the context you send

Caching reuses context cheaply, but the cheapest token is the one you never send. Audit what's actually in the window:

- **Prune the system prompt.** Long instruction blocks accumulate cruft. Cut examples that no longer earn their tokens; keep one strong example over five mediocre ones.
- **Retrieve, don't dump.** Instead of pasting an entire document, fetch only the relevant passages ([RAG](/docs/foundations/rag)). Sending a 50-page PDF to answer one question is the most common waste.
- **Compact long sessions.** When a conversation grows, replace old turns with a short running summary instead of carrying every message forever. The history is input tokens you re-pay on every call.
- **Right-size the tool catalog.** Each tool definition is input tokens on every request. Expose only the tools the current task needs.
- **Compress static memory files.** [`CLAUDE.md`](/docs/claude-code/claude-md) and memory notes are reused-input you re-pay every session. Rewriting them in terse "caveman" prose (e.g. the `caveman-compress` command from the [`caveman`](https://github.com/JuliusBrussee/caveman) plugin) trims a slice — but expect single digits on files that are already lean: a hands-on pass over a tight ~350-word `CLAUDE.md` cut only ~10%. The lever is real but small; don't sacrifice instruction clarity in your global config to chase it.

## Right-size the model

Don't pay Opus rates for a Haiku-grade task. Classification, extraction, simple formatting, and routing usually run great on the smallest model at a fraction of the per-token price. Reserve the larger models for genuinely hard reasoning, and consider routing: a cheap model handles the easy majority, escalating only the hard cases. See [Choosing a Model](/docs/api/choosing-a-model) and [Tokens, Context & Pricing](/docs/api/tokens-and-pricing) for the tradeoffs.

## Prefer structured output over prose

Asking for JSON (or another tight schema) instead of an explanatory paragraph cuts output tokens *and* removes the parsing guesswork downstream. Telling Claude to return only a compact object like `{"label": ..., "score": ...}` generates a fraction of the tokens of a chatty answer — and you skip the "Here's the result:" preamble entirely. Details in [Structured Output](/docs/api/structured-output).

## Batch what isn't time-sensitive

For offline work where you don't need an answer in seconds — evals, bulk classification, dataset labeling, summarizing an archive — Anthropic's **Message Batches API** runs requests asynchronously at a **50% discount on both input and output tokens**, with results typically returned within 24 hours.

<VerifyNote lastVerified="2026-06-24" source="https://www.anthropic.com/news/message-batches-api">
The Message Batches API advertises a flat 50% discount versus the synchronous Messages API and async (up to ~24h) turnaround. Per-batch request limits and pricing change — confirm current figures in Anthropic's docs before relying on them.
</VerifyNote>

Stack this with caching and a right-sized model and the combined discount on a large offline job is dramatic.

## Measure — don't guess

Optimize against numbers, not vibes. Anthropic's **token-counting endpoint** returns the exact input-token count for a request *before* you send it — same shape as a Messages call, and it's free (rate-limited). Use it to compare a bloated prompt against a trimmed one, to make model-routing decisions, and to keep prompts inside the context window.

<PromptCard title="Count tokens before sending (Python SDK)">
{`import anthropic

client = anthropic.Anthropic()

resp = client.messages.count_tokens(
    model="claude-opus-4-8",
    system="You are a scientist",
    messages=[{"role": "user", "content": "Hello, Claude"}],
)
print(resp.input_tokens)  # exact input count, no charge for counting`}
</PromptCard>

<Callout type="tip" items={["Don't use another model's tokenizer (e.g. tiktoken) — counts differ per model family.
