Du hast einen Prompt angepasst und er *fühlt* sich besser an — aber stimmt das? Ohne **Evals** (Evaluierungen) fliegst du blind: Jede Änderung ist ein Münzwurf, und du erfährst von einem verärgerten Nutzer, dass etwas kaputt ist, nicht von einem Test. Evals verwandeln "Bauchgefühl" in eine Zahl, der du vertrauen, die du verteidigen und über die Zeit beobachten kannst. Das ist der größte einzelne Unterschied zwischen Hobby-Prompts und produktionsreifer Claude-Arbeit.

<Callout type="objectives" items={[

Eval-Tooling und Modellverhalten bewegen sich schnell — prüfe die verlinkten Guides erneut und **kalibriere deinen Judge neu, sobald du das zu testende Modell änderst**. OWASP listet das Falschmachen davon (sowie Prompt Injection) weiterhin unter den größten LLM-Risiken für 2026.
</VerifyNote>

## Die Denkweise: messen, nicht raten

Drei Regeln, die dich retten:

- **Bottom-up schlägt top-down.** Sammle zuerst *tatsächliche* Fehler und entwirf dann die Metrik, um sie abzufangen. Ein Eval, das aus echten Pannen entsteht, sagt echte Pannen voraus; ein Eval, das am Whiteboard erfunden wurde, misst meist nur deine Vorstellungskraft.
- **Eine Zahl, die du erneut ausführen kannst.** Ein Eval ist wiederholbar: gleiche Eingaben → vergleichbarer Score. Genau das erlaubt dir, Prompt v1 vs. v2 oder `claude-haiku-4-5` vs. `claude-sonnet-4-6` ehrlich zu vergleichen.
- **Günstig auszuführen, oft auszuführen.** Wenn ein Mensch dafür einen Nachmittag braucht, passiert es nicht. Automatisiere es.

## Baue einen Golden Dataset (bottom-up)

Dein Golden Dataset ist das Herz jedes Evals — eine kuratierte Menge an Eingaben mit bekannt-guten Erwartungen.

<Steps items={[
  {title: "Sammle zuerst echte Fehler", body: "Beginne mit tatsächlich schlechten Ausgaben: Produktions-Traces, Bug-Reports, Support-Tickets. Das sind die Fälle, die zählen."},
  {title: "Verankere mit 50–100 handgefertigten Fällen", body: "Schreibe von Hand Fälle, die deine kritischsten und fehleranfälligsten Szenarien abdecken. Das ist dein stabiler Anker-Set."},
  {title: "Erweitere Richtung ~500+", body: "Füge anonymisierte Produktionsbeispiele (entferne PII) und synthetische Fälle für unterrepräsentierte Szenarien hinzu. Vertraue keinen Aggregat-Metriken auf einem winzigen Set."},
  {title: "Lass ihn jede Woche wachsen", body: "Jede neue Produktionsregression wird zu einem neuen Testfall. Ein Golden Dataset lebt, er ist nicht eingefroren."},
]} />

## Bewerten: zuerst Code, dann Judge

Greife zuerst zur günstigsten zuverlässigen Prüfung.

- **Programmatische (deterministische) Prüfungen** — nutze diese überall dort, wo die Antwort eine Struktur hat: exakter/Keyword-Abgleich, "gültiges JSON gegen dieses Schema", "hat es das richtige Tool mit den richtigen Argumenten aufgerufen", "unter N Tokens / unter X ms". Schnell, kostenlos und nie unzuverlässig.
- **LLM-as-Judge** — für unscharfe Dimensionen (Hilfreichkeit, Tonfall, Treue zu einer Quelle), die sich Code widersetzen. Gib dem Judge eine **Rubrik**, kein Bauchgefühl, und **kalibriere ihn gegen menschliche Labels**, bevor du ihm vertraust.

:::warning Judges haben Verzerrungen
LLM-Judges driften zu längeren Antworten (Verbosity Bias) und zu der Option, die zuerst gezeigt wird (Position Bias). Gegenmaßnahmen: eine strenge Rubrik, paarweiser Vergleich statt absoluter Bewertung, Vertauschen der Antwortreihenfolge und erneutes Prüfen des Judges gegen einen menschlich gelabelten Ausschnitt. Ein Judge ist **eine Schicht**, nicht der gesamte Test.
:::

## Bei Agenten teste auch die *Trajektorie*

Ein Agent kann die richtige Endantwort auf dem falschen Weg erreichen — in Schleifen laufen, ein destruktives Tool aufrufen oder dein Budget verbrennen. Bewerte also den Weg, nicht nur das Ziel: **Hat es die richtigen Tools in einer sinnvollen Reihenfolge aufgerufen, ohne Schleifen, innerhalb des Budgets?** Tool-Call-Korrektheit und Trajektorien-Prüfungen fangen Fehler ab, die ein Eval, das nur die Endantwort betrachtet, niemals sieht.

## Verdrahte es in die CI

Hier zahlen sich Evals aus: Mache Regressionen unmöglich zu mergen.

<Steps items={[
  {title: "Führe den Golden Set bei jedem Pull Request aus", body: "Bewerte programmatisch, wo möglich; lass den Judge auf den Rest los."},
  {title: "Lass den Build bei einem Abfall fehlschlagen", body: "Setze eine Schwelle (z. B. der Score darf nicht gegenüber main sinken). Eine Prompt-Änderung, die die Qualität verschlechtert, kann nicht ausgeliefert werden."},
  {title: "Schließe den Kreis in der Produktion", body: "Wenn ein Judge eine Live-Antwort markiert, leite sie in eine menschliche Review-Queue; der Reviewer bestätigt, fügt den Fall dem Golden Set hinzu und testet nach dem Fix erneut."},
]} />
