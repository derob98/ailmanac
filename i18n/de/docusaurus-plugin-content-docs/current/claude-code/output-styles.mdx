**Output-Styles** ändern, *wie* Claude Code kommuniziert — seine Rolle, seinen Tonfall, seine Ausführlichkeit und ob es seine Überlegungen erklärt — indem sie den System-Prompt direkt anpassen. Sie ändern, *wie* Claude antwortet, nicht *was* es weiß. Greife zu einem, wenn du jede Runde aufs Neue denselben Tonfall oder dasselbe Format anforderst.

## Warum sie nützlich sind

- **Passend zu deinem Workflow.** Knapp und handlungsorientiert, wenn du schnell vorankommen willst; erklärend, wenn du gerade lernst.
- **Lehrmodus.** Manche Styles fügen lehrreiche Kommentare hinzu — ideal, während du dich mit einer Codebasis oder einer Technik vertraut machst.
- **Konsistenz.** Lege einmal einen bevorzugten Style fest, damit sich jede Sitzung so liest, wie du es magst.

## Die integrierten Styles

Claude Code wird mit vier ausgeliefert:

| Style | Was er bewirkt |
|---|---|
| **Default** | Der standardmäßige System-Prompt für Softwareentwicklung — effizient, handlungsorientiert. |
| **Proactive** | Führt sofort aus und trifft sinnvolle Annahmen, anstatt bei Routineentscheidungen innezuhalten. Stärkere Anleitung zur autonomen Ausführung als der Auto-Modus, aber du siehst weiterhin Berechtigungsabfragen, bevor Tools ausgeführt werden. |
| **Explanatory** | Fügt lehrreiche "Insights" zwischen den Schritten ein und erklärt Implementierungsentscheidungen und Muster der Codebasis. |
| **Learning** | Kollaborativ, learn-by-doing: teilt Insights *und* bittet dich, kleine Teile selbst zu schreiben, wobei `[TODO(human)]`-Markierungen in deinem Code hinterlassen werden. |

Explanatory und Learning erzeugen bewusst längere Antworten und kosten daher mehr Output-Tokens als Default.

## So wechselst du

Führe `/config` aus und wähle einen Style unter **Output style**. Deine Auswahl wird auf Projektebene in `.claude/settings.local.json` gespeichert. Um sie ohne das Menü festzulegen, bearbeite das Feld `outputStyle` in einer beliebigen Einstellungsdatei:

```json
{
  "outputStyle": "Explanatory"
}
```

:::warning Änderungen erfordern eine neue Sitzung
Der Output-Style ist Teil des System-Prompts, den Claude Code **einmal beim Sitzungsstart** liest. Eine Änderung wird erst nach `/clear` oder in einer neuen Sitzung wirksam. (Der eigenständige Befehl `/output-style` wurde in v2.1.91 entfernt — verwende `/config` oder die Einstellung oben.)
:::

## Einen eigenen Style erstellen

Ein eigener Style ist eine Markdown-Datei: Frontmatter, gefolgt von den Anweisungen, die an den System-Prompt angehängt werden. Speichere sie auf Benutzerebene (`~/.claude/output-styles/`) oder auf Projektebene (`.claude/output-styles/`); der Dateiname wird zum Namen des Styles, sofern du nicht `name` setzt.

```markdown
---
name: Diagrams first
description: Lead every explanation with a diagram
keep-coding-instructions: true
---

When explaining code, architecture, or data flow, start with a Mermaid
diagram showing the structure, then explain in prose.
```

Setze `keep-coding-instructions: true`, wenn du nur änderst, *wie* Claude kommuniziert, aber weiterhin möchtest, dass es normal programmiert. Lasse es weg (der Standard), wenn Claude überhaupt keine Softwareentwicklung betreibt — etwa als reiner Schreib- oder Datenanalyse-Assistent —, damit die integrierten Programmieranweisungen verworfen werden.

## Output-Styles vs. CLAUDE.md vs. andere Hebel

- **Output-Style** = eine System-Prompt-Voreinstellung für Rolle, Tonfall und Format — gilt für *jede* Antwort.
- **[CLAUDE.md](/docs/claude-code/claude-md)** = dauerhafte *Projekt*-Fakten und -Regeln (Konventionen, Leitplanken), die als Nachricht nach dem System-Prompt hinzugefügt werden.
- **[Eigene Anweisungen / Antwort-Styles](/docs/claude-app/custom-instructions)** = das Pendant in der Claude.ai-Endkunden-App.

:::tip Wähle den richtigen Hebel
Möchtest du, dass Claude "vor dem Abschluss immer Tests ausführt"? Das ist eine **CLAUDE.md-Leitplanke**, kein Style. Möchtest du, dass es "jeden Schritt unterwegs erklärt"? Das ist ein **Output-Style**. Halte Verhaltensregeln in CLAUDE.md und die Darstellung in Styles.
:::
