एक बार जब आप हाथ से कुछ एजेंट बना लेते हैं, तो वही प्लंबिंग बार-बार दिखाई देती है: एक लूप जो मॉडल को कॉल करता है, उसने जो टूल माँगा उसे चलाता है, परिणाम वापस फ़ीड करता है, और जब काम पूरा हो जाता है तो रुक जाता है। **एजेंट फ्रेमवर्क** उस प्लंबिंग को पैकेज कर देते हैं — साथ ही स्टेट, मेमोरी और मल्टी-एजेंट समन्वय — ताकि आपको कम ग्लू कोड लिखना पड़े। यह पेज मुख्य **ओपन-सोर्स** विकल्पों का एक टिकाऊ, प्रोवाइडर-न्यूट्रल नक्शा है, वे वास्तव में आपको क्या देते हैं, और कैसे चुनें। इनमें से लगभग सभी **मॉडल-एग्नॉस्टिक** हैं: वे Claude, GPT, Gemini और लोकल ओपन-वेट मॉडल के साथ काम करते हैं।

## एक एजेंट फ्रेमवर्क वास्तव में आपको क्या देता है

ब्रांडिंग को हटा दें तो एक फ्रेमवर्क आपको चार चीज़ों में से कुछ उपसमूह पेश कर रहा है:

- **ऑर्केस्ट्रेशन** — कंट्रोल लूप। क्रमिक चरण, शाखाएँ, पुनः प्रयास, लूप-जब-तक-पूरा-न-हो, और (बढ़ती हुई मात्रा में) **ड्यूरेबल एग्ज़ीक्यूशन** जो क्रैश से बच जाता है और वहीं से फिर शुरू होता है जहाँ रुका था।
- **टूल्स** — एक ऐसे फ़ंक्शन को घोषित करने का एक मानक तरीका जिसे मॉडल कॉल कर सकता है, उसके आर्गुमेंट को वैलिडेट करता है, उसे चलाता है, और परिणाम लौटाता है। वही describe→call→execute→return लूप जिसे आप [टूल यूज़](/docs/api/tool-use) से पहले से जानते हैं।
- **मेमोरी और स्टेट** — बातचीत का इतिहास, स्क्रैचपैड तथ्य, और पुनर्प्राप्त किए गए दस्तावेज़ टर्न-दर-टर्न रखने के लिए कोई जगह, ताकि एजेंट चरणों के बीच स्मृति-रहित न हो।
- **मल्टी-एजेंट** — कई विशेषज्ञ एजेंटों के पैटर्न जो एक-दूसरे को काम सौंपते हैं: एक सुपरवाइज़र जो वर्कर्स को काम सौंपता है, या साथी जो किसी काम पर सहयोग करते हैं। (अवधारणात्मक रूप से Claude Code के [सबएजेंट](/docs/claude-code/subagents) जैसा ही विचार।)

एक फ्रेमवर्क तब सार्थक है जब अन्यथा आपको इन चारों को हाथ से लिखना और मेंटेन करना पड़ता। यह तब सार्थक *नहीं* है जब आपका काम "मॉडल को कॉल करो, एक-दो टूल चलाओ, जवाब लौटाओ" हो — वहाँ फ्रेमवर्क एक ओवरहेड है जिससे लड़ने में आप समय बर्बाद करेंगे।

## तीन आर्किटाइप

फ्रेमवर्क उतने भिन्न नहीं होते जितना उनकी मार्केटिंग सुझाती है। वास्तव में तीन ही आकार हैं, और अधिकांश परियोजनाएँ इनमें से किसी एक का रूपांतर हैं:

- **स्टेटफुल ग्राफ / वर्कफ़्लो।** आप एजेंट को नोड्स और एजेस (या चरण और ट्रांज़िशन) के एक स्पष्ट ग्राफ के रूप में मॉडल करते हैं। अधिकतम नियंत्रण, निरीक्षण-योग्य स्टेट, लंबे समय तक चलने वाले और ह्यूमन-इन-द-लूप फ्लो के लिए अच्छा। शुरुआत में सीखने को ज़्यादा। → **LangGraph**, **LlamaIndex Workflows**।
- **रोल-आधारित क्रू।** आप एजेंटों का वर्णन *भूमिका* के अनुसार करते हैं ("शोधकर्ता", "लेखक", "समीक्षक") और उन्हें सहयोग करने या एक प्रक्रिया चलाने देते हैं। एक मल्टी-एजेंट टीम को व्यक्त करने में तेज़; आप उच्च-स्तरीय अमूर्तन के बदले कुछ सूक्ष्म नियंत्रण का व्यापार करते हैं। → **CrewAI**, और **AutoGen** की संवादात्मक मल्टी-एजेंट शैली।
- **मिनिमल लूप / कम अमूर्तन।** मॉडल की मूल टूल-कॉलिंग पर एक पतली परत, जिसमें कुछ एजेंटों के बीच हैंडऑफ़ हों और बहुत कुछ नहीं। शुरू से अंत तक पढ़ने में आसान, हटाने में आसान। → **OpenAI Agents SDK** (प्रायोगिक **Swarm** का प्रोडक्शन उत्तराधिकारी), और **आपके खुद के लिखे सादे लूप**।

## एक त्वरित दौरा (प्रतिबद्ध होने से पहले पोज़िशनिंग की पुष्टि करें)

ये जानने योग्य ओपन परियोजनाएँ हैं। नाम केवल इसलिए दिए गए क्योंकि प्रत्येक का असली repo सत्यापित किया गया था; हर अस्थिर चीज़ ऊपर दिए गए VerifyNote के पीछे रहती है।

- **LangGraph** — ग्राफ के रूप में मॉडल किए गए स्टेटफुल, लंबे समय तक चलने वाले एजेंटों के लिए एक निम्न-स्तरीय ऑर्केस्ट्रेशन फ्रेमवर्क; ड्यूरेबल एग्ज़ीक्यूशन और ह्यूमन-इन-द-लूप प्रथम-श्रेणी के हैं। स्टैंडअलोन या व्यापक **LangChain** पारिस्थितिकी तंत्र के साथ उपयोग करने योग्य। मॉडल-एग्नॉस्टिक। <VerifyNote lastVerified="2026-06-28" source="https://github.com/langchain-ai/langgraph" />
- **LlamaIndex** — एक डेटा/RAG फ्रेमवर्क के रूप में शुरू हुआ (कनेक्टर, इंडेक्स, रिट्रीवल) और अब एजेंटों के लिए एक इवेंट-चालित **Workflows** परत भी देता है। तब मज़बूत जब आपका एजेंट मूल रूप से *आपके दस्तावेज़ों पर रिट्रीवल* हो। <VerifyNote lastVerified="2026-06-28" source="https://github.com/run-llama/llama_index" />
- **Microsoft AutoGen** — संवादात्मक मल्टी-एजेंट सिस्टम के लिए एक फ्रेमवर्क। 2026 के मध्य तक यह **मेंटेनेंस मोड** में है; Microsoft नई परियोजनाओं को एक एकीकृत उत्तराधिकारी (**Microsoft Agent Framework**, AutoGen + Semantic Kernel का विलय) की ओर निर्देशित करता है। शुरू करने से पहले वर्तमान स्थिति जाँचें। <VerifyNote lastVerified="2026-06-28" source="https://github.com/microsoft/autogen" />
- **CrewAI** — एक रोल-आधारित फ्रेमवर्क: एजेंटों को भूमिका और लक्ष्य के अनुसार परिभाषित करें, उन्हें **Crews** (स्वायत्त सहयोग) या **Flows** (इवेंट-चालित नियंत्रण) में व्यवस्थित करें। एक मल्टी-एजेंट टीम तक तेज़ रास्ता। <VerifyNote lastVerified="2026-06-28" source="https://github.com/crewAIInc/crewAI" />
- **OpenAI Agents SDK** — हैंडऑफ़ के साथ मल्टी-एजेंट वर्कफ़्लो के लिए एक जानबूझकर हल्का, कम-अमूर्तन वाला फ्रेमवर्क। नाम के बावजूद यह **प्रोवाइडर-एग्नॉस्टिक** है (इसके docs 100+ LLM के समर्थन का उल्लेख करते हैं), और यह प्रायोगिक **Swarm** का प्रोडक्शन-रेडी उत्तराधिकारी है। <VerifyNote lastVerified="2026-06-28" source="https://github.com/openai/openai-agents-python" />
- **सादा एजेंट लूप** — कोई फ्रेमवर्क बिल्कुल नहीं: मॉडल की मूल टूल-कॉलिंग के इर्द-गिर्द आपका खुद का `while` लूप। सरल एजेंटों के लिए सही डिफ़ॉल्ट, और वह चीज़ जिसे ऊपर का हर फ्रेमवर्क अंततः लपेट रहा है।

## वह चीज़ जिसे हर फ्रेमवर्क लपेटता है

किसी भी लाइब्रेरी तक पहुँचने से पहले, उस लूप को देखना मददगार होता है जिस पर वे सभी बने हैं। यही पूरा विचार है — मॉडल तय करता है, आप निष्पादित करते हैं, तब तक दोहराते हैं जब तक काम पूरा न हो जाए:

```python
# Provider-neutral agent loop — the core every framework wraps.
# `model_call` and `run_tool` are yours; swap in Claude, GPT, Gemini, or a local model.

def agent_loop(task, tools, max_steps=10):
    messages = [{"role": "user", "content": task}]

    for _ in range(max_steps):
        # 1. Ask the model what to do next (it sees the tool schemas).
        response = model_call(messages, tools=tools)

        # 2. No tool requested → the model is done. Return its answer.
        if not response.tool_calls:
            return response.text

        # 3. Run each requested tool and feed results back in.
        messages.append(response.as_message())
        for call in response.tool_calls:
            result = run_tool(call.name, call.arguments)
            messages.append({
                "role": "tool",
                "tool_call_id": call.id,
                "content": result,
            })

    return "Stopped: hit max_steps without finishing."
```

अगर आप इसे पढ़ सकते हैं, तो आप समझते हैं कि इस पेज पर मौजूद हर फ्रेमवर्क अंदरूनी तौर पर क्या कर रहा है। ग्राफ स्पष्ट स्टेट और शाखा जोड़ते हैं; क्रू भूमिकाएँ और प्रत्यायोजन जोड़ते हैं; मिनिमल SDK सुव्यवस्थित हैंडऑफ़ जोड़ते हैं — लेकिन दिल की धड़कन हमेशा यही लूप है।

- [LlamaIndex — GitHub](https://github.com/run-llama/llama_index) — एजेंटों के लिए इवेंट-चालित Workflows परत वाला डेटा/RAG फ्रेमवर्क।
- [LlamaIndex docs](https://developers.llamaindex.ai/) — RAG, क्वेरी इंजन, एजेंट, और Workflows।
- [Microsoft AutoGen — GitHub](https://github.com/microsoft/autogen) — संवादात्मक मल्टी-एजेंट फ्रेमवर्क (मेंटेनेंस स्थिति जाँचें)।
- [Microsoft Agent Framework — GitHub](https://github.com/microsoft/agent-framework) — एजेंट बनाने और ऑर्केस्ट्रेट करने के लिए एकीकृत उत्तराधिकारी (AutoGen + Semantic Kernel)।
- [CrewAI — GitHub](https://github.com/crewAIInc/crewAI) — Crews और Flows वाला रोल-आधारित मल्टी-एजेंट फ्रेमवर्क।
- [CrewAI docs](https://docs.crewai.com/) — एजेंट, टास्क, क्रू, फ्लो, और टूल।
- [OpenAI Agents SDK — GitHub](https://github.com/openai/openai-agents-python) — हैंडऑफ़ के साथ हल्का, प्रोवाइडर-एग्नॉस्टिक मल्टी-एजेंट फ्रेमवर्क।

- [OpenAI Swarm — GitHub](https://github.com/openai/swarm) — प्रायोगिक पूर्ववर्ती जिसे Agents SDK प्रतिस्थापित करता है (शैक्षिक, अब अधिक्रमित)।
