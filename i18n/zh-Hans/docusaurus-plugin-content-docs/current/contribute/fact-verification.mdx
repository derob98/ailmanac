“始终最新”是我们靠一套简单、诚实的体系来兑现的承诺——而非魔法。下面介绍你在贡献时如何守住这个承诺。

## 长青内容 vs 易变内容

- **长青内容** — 不会变化的概念（什么是 LLM、什么是 RAG）。**无需**日期。
- **易变内容** — 任何会随版本变化的内容：模型名称/ID、价格、限制、UI 标签、测试版/功能可用性、确切的 CLI 参数。**必须带上日期 + 来源。**

## 为易变事实盖戳

在级别徽章正下方使用 `<VerifyNote>` 组件：

```mdx
<VerifyNote lastVerified="2026-06-20" source="https://docs.anthropic.com/...">
One line on what's volatile here and to confirm at the source.
</VerifyNote>
```

- **`lastVerified`** = *你*实际对照来源核实它的日期。
- **`source`** = 具有权威性的官方页面。

## 切勿硬编码模型相关事实

模型 ID/定价/上下文**只**存放在 `data/models.json` 中，由[模型页面](/docs/whats-new/models-and-pricing)上的 `<ModelTable />` 渲染。在正文里，请**链接到那里**，而不要写出某个模型 ID 或价格。更新模型 = 对那个 JSON 做一行编辑（一个完美的首个 PR）。

## 响应 `needs-verification`

当某个事实可能已过时，自动化（以及读者）会开标记为 `needs-verification` 的 issue。要解决它：

1. 对照官方来源核对该说法。
2. 若正确 → 把 `lastVerified` 更新为今天。
3. 若已变化 → 既修正内容**也**更新日期，并链接来源。
4. 若是模型相关事实 → 更新 `data/models.json`，而非正文。

## 诚实原则

:::tip 一个旧日期是一项特性
来自数月前的 `lastVerified` 并不意味着我们失职了——它是在告诉读者去复核*那一项*事实。诚实的旧日期，胜过自信的错误内容。不要更新一个你并未真正重新核实过的日期。
:::
