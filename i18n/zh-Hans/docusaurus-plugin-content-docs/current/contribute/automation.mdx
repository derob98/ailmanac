透明度页面：这里展示了帮助 AILmanac 以最少维护者投入保持内容新鲜、高质量的机制。贯穿始终的原则是：**自动化只是把工作呈现出来供人类审批——它从不悄悄发布未经核实的内容。**

## 构建门禁（已启用）

每一次改动都会以 `onBrokenLinks: 'throw'` 的方式构建，因此失效的内部链接或损坏的 MDX 都会**在合并前让构建失败**。这是最有价值的一项检查——它让导航始终值得信赖。

## CI 工作流

> 工作流文件位于 `ci-workflows` 分支，当仓库拥有具备 `workflow` 权限范围的令牌后即可启用（参见 README）。在此之前，站点从 `gh-pages` 分支部署。

- **`build-check`** — 构建每一个 PR（链接失效则失败）。这是一项必需检查。
- **`deploy`** — 在推送到 `main` 时构建并发布到 GitHub Pages。
- **`link-check`** — `lychee` 每周以及在 PR 上运行；遇到失效的外部链接时，它会**开一个跟踪 issue**（标签为 `link-rot`），而不是只是变红。

## 新鲜度自动化（规划中／分阶段）

- **发布监视器** — 一个定时任务，轮询 Anthropic 的文档/更新日志**以及**变动最快的消费端界面（应用、语音、移动端、市场）。一旦发生变化，它会开一个标记为 `upstream-change` + `needs-verification` 的 issue/PR，指向可能受影响的页面。由人类来撰写更新。
- **模型表校验器** — 将 `data/models.json` 与官方模型列表做差异比对，发生漂移时开一个 PR。[模型表](/docs/whats-new/models-and-pricing)会显示一个“最近核实”日期。
- **陈旧度巡检** — 标记那些 `lastVerified` 早于阈值的易变页面。

## 为什么要人类参与其中

:::tip 自动化且诚实 > 自动化但出错
一个会自动把“新闻”提交到参考站点的 AI 有发布幻觉内容的风险——这与值得信赖恰恰相反。所以我们的自动化只**提议**（issue/PR）；由人类来**核实并合并**。参见[事实核实](/docs/contribute/fact-verification)。
:::
