你已经习惯用 Claude 的方式思考。可有一天，一位同事对 **Grok** 回答"*此时此刻*正在发生什么"类问题的能力赞不绝口；某个项目需要一个直接接入 X/Twitter 的助手；或者有人扔给你一份 OpenAI 形态的代码库，让你把它指向 xAI。好消息是：你所知道的几乎一切都能迁移过去——而且在某一点上（API）切换几乎是零成本的。本页把你的 Claude 心智模型映射到 **xAI 的 Grok** 上，标出那少数几个真正运作方式不同的地方，并告诉你什么时候 Grok 往往是更好的选择。

## 60 秒概念地图

如果你只读一节，就读这节。下面是你在 Claude 里熟知的东西，与 Grok 世界的对应关系：

| 在 Claude 里你叫它…… | 在 Grok 的世界里它是…… | 同一个概念吗？ |
| --- | --- | --- |
| Claude.ai（聊天应用） | **Grok 应用**（grok.com、X 应用、移动端） | 是——消费级/助手界面 |
| 模型选择器（Opus / Sonnet / Haiku） | Grok 的各种变体（旗舰款如 **Grok 4.3**、一个 **fast** 工具调用变体、一个 **coding** 编码变体） | 是——同样的概念，不同的名字 |
| Claude Code（终端里的智能体编码） | **Grok Build**（xAI 的编码智能体 CLI） | 相近——见下文 |
| Console / API + 工作台 | **xAI API**（`https://api.x.ai/v1`）+ xAI 控制台 | 是——用来原型开发和拿密钥的开发者界面 |
| 工具使用 / 函数调用（API） | **函数调用**（兼容 OpenAI 的形态） | 是——同样的循环，而且这里连 JSON 都很眼熟 |
| 网页搜索 / 引用 | 内置于模型的**原生实时 X + 网页搜索** | 相近——但明显更"实时" |

要点：几乎没有*新概念*要学。它主要是把你早已理解的东西**改名和挪位**——外加两个 Grok 确实比 Claude 走得更远的领域：**对 X/网页数据的实时访问**，以及**一键式、兼容 OpenAI 的 API**。

## API 可一键接入——这对开发者是重头戏

这里有一个对开发者改变最大的差异：**xAI API 遵循 OpenAI 的 API 规范。** 基础 URL 是 `https://api.x.ai/v1`，所以如果你已经有代码在跟一个兼容 OpenAI 的客户端对话，迁移往往就是*改一下基础 URL、换一下密钥*。<VerifyNote lastVerified="2026-07-01" source="https://docs.x.ai/developers/quickstart">xAI 将其 API 记录为兼容 OpenAI，基础 URL 为 `https://api.x.ai/v1`；确切的兼容范围（哪些参数和端点能干净地对应）会不断演进——请对照 xAI 当前的快速上手文档核实。</VerifyNote>

对于 Claude 用户，这一点值得内化，因为 Claude 的 Messages API 有*自己的*形态（自己的请求/响应格式，自己的 `tool_use`/`tool_result` 块）。从 Claude 过来，请这样调整你的条件反射：

<Steps items={[
  {title: "指向 OpenAI SDK，而不是 Anthropic 的", body: "Grok 是通过 OpenAI 风格的客户端来访问的。你不用翻译成第三种方言——你复用你（或某个库）早已熟悉的 OpenAI 请求形态。"},
  {title: "改基础 URL + 密钥", body: "把基础 URL 设为 https://api.x.ai/v1，并使用 xAI 密钥。你现有的大部分兼容 OpenAI 的管道都能继续工作。"},
  {title: "几乎逐字复用工具调用 JSON", body: "因为格式兼容 OpenAI，函数调用定义的迁移比在 Claude 和 OpenAI 之间要直接得多——这是一个实实在在的便利，而不是陷阱。"},
  {title: "仍要确认模型 ID", body: "你必须正确设置的一件事就是模型名称（例如某个当前的 grok-4.x id）。模型 ID 是变动最快的细节——查一下实时的模型页面。"},
]} />

## 实时 X + 网页搜索才是真正的头条

这里是 Grok 与你的 Claude 习惯分歧最大的地方。Grok 由 xAI 打造，具备**对 X（Twitter）和网页信息的原生、实时访问**，xAI 把实时搜索定位为模型作答方式中的一等公民。<VerifyNote lastVerified="2026-07-01" source="https://x.ai/news/grok-4">xAI 将 Grok 描述为具备原生工具使用和实时搜索集成（包括来自 X 的数据）；确切的检索行为、新鲜度以及任何按套餐的限制都会变化——请在 xAI 当前的产品页面上核实。</VerifyNote>

对于 Claude 用户，心智上的转变是这样的：用 Claude 时，当你需要*当前*信息，你会伸手去拿一个搜索工具、一条检索管线或一个连接器，并且要谨慎地推敲你的知识截止日期。而用 Grok，"此刻 X 上/网上正在发生什么"更接近于一个内置默认能力。这使得 Grok 成为**实时、时事和社交脉搏**类问题的天然选择——而这也改变了你应该*不信任*什么：实时结果仍然需要你对任何网页内容都会施加的同样的溯源核查纪律。

<Callout type="tip" items={[
  "在新鲜度这个维度上依靠 Grok：突发新闻、热门讨论、'今天大家在说什么'。这在结构上就是它的强项。",
  "在验证上保留你的 Claude 直觉：实时 ≠ 权威。索要来源，把一条爆款帖子当作待核查的主张，而不是事实。",
  "新鲜度有半衰期。对于经久耐用的参考资料，一个实时搜索的答案可能会过时，或者过度偏向最近声音最大的那种论调——见下面的说明。",
]} />

要更深入地理解*为什么*"新近"和"真实"不是一回事，请见 [新鲜度与半衰期](/docs/frontiers/freshness-half-life) 和 [幻觉](/docs/foundations/hallucinations)——两者都直接适用于任何实时搜索助手。

## Grok Build 是 xAI 对 Claude Code 的回应

如果你活在 **Claude Code** 里，xAI 的 **Grok Build** 就是最接近的对应物：一个你从终端里驱动的编码智能体，与 Claude Code 和 OpenAI 的 Codex CLI 处在同一赛道上竞争。它围绕一个针对编码调优的 Grok 变体构建，并采用并行子智能体的方式来处理较大代码库，还带有一个用于交接更大任务的长时自主模式。<VerifyNote lastVerified="2026-07-01" source="https://docs.x.ai/developers/models">xAI 已经发布了一个编码智能体 CLI（Grok Build）和一个针对编码调优的模型变体；可用性（例如哪个订阅层级）、确切的功能集以及模型 id 变动很快——请把具体细节视为时效性内容并加以核实。</VerifyNote>

这些概念从 Claude Code 几乎一一对应地迁移过来：

| Claude Code 的概念 | Grok Build 的对应物 | 说明 |
| --- | --- | --- |
| 从终端进行智能体式编辑 | 相同——一个 CLI 编码智能体 | 核心循环（读 → 规划 → 编辑 → 运行）是整个这一品类共有的形态 |
| 子智能体 / 并行工作 | 并行子智能体架构 | 两者都把大工作拆给助手；编排细节不同 |
| 长时自主任务 | 一个长时的"把目标做完"模式 | 交出一个更大的实现，让它规划-执行-验证 |

经久的要点：**驱动一个智能体式编码 CLI 的技能是可移植的。** 写一份清晰的任务、把上下文保持精简、审阅 diff、用检查点来把控方向——无论智能体是 Claude Code、Codex 还是 Grok Build，这些都是同样的纪律。关于心智模型见 [什么是 Claude Code？](/docs/claude-code/what-is-claude-code)，关于更广阔的领域见 [本地编码智能体](/docs/models/local-coding-agents)。

## 真正需要调整的地方

你的大部分 Claude 经验都能迁移。下面是那份真正有所不同的简短清单：

- **API 方言（对你有利）。** Grok 说的是兼容 OpenAI，而不是 Anthropic Messages。如果你从 Claude 的 API 过来，你就切换到 OpenAI 的请求/响应形态——并且能轻松复用任何已经为兼容 OpenAI 端点构建的东西。
- **默认实时。** 时事扎根在 Grok 那里更像是内置能力。调整你何时伸手去用它——并且要持续核查来源。
- **人设与边界。** Grok 的口吻和内容边界与 Claude 的调法不同。别以为一个为 Claude 风格校准过的系统提示词会产生同样的语气或同样的拒答。
- **易变的具体项。** 模型名称、变体、上下文限制和价格变动很快（Grok 在 2026 年发布了多个版本）。养成在源头确认的习惯——就像你对 Claude 也应该做的那样。

## Grok 往往在何时脱颖而出

两者都是能干的通用助手，在任何给定任务上的差距往往很小且在不断变化——所以把这当作*倾向*，而非定律，让你自己[在你自己的数据上做的评测](/docs/models/choosing-a-model)来决定：

- **实时与时事类工作。** "此刻正在发生什么"、热门话题以及社交脉搏类问题，正中 Grok 原生实时 X/网页搜索的下怀。
- **一键式 OpenAI 迁移。** 如果你已经有兼容 OpenAI 的代码，把它指向 xAI 是这个领域里摩擦最低的切换之一。
- **终端优先的编码交接。** 如果你喜欢 Claude Code / Codex 的工作流，Grok Build 是一个天然的第三选项，可以在你自己的任务上做 A/B。

大多数情况下最诚实的答案是：**你（或你的团队）已经搭建好的那一个，以及在真正要紧的那个任务上赢得你评测的那一个。** 技能是可移植的；搭建成本才是真正的切换成本。关于与厂商无关的选择方法，见 [选择一个模型](/docs/models/choosing-a-model) 和 [各家厂商的 AI 成本几何](/docs/models/what-ai-costs-across-providers)。
