"Sempre atualizado" é uma promessa que mantemos com um sistema simples e honesto — não com mágica. Veja como cumpri-la quando você contribui.

## Perene vs. volátil

- **Perene** — conceitos que não mudam (o que é um LLM, o que é RAG). **Nenhuma data** necessária.
- **Volátil** — qualquer coisa que mude de release para release: nomes/IDs de modelos, preços, limites, rótulos de UI, disponibilidade de beta/funcionalidade, flags exatas de CLI. **Deve carregar uma data + fonte.**

## Carimbe os fatos voláteis

Use o componente `<VerifyNote>` logo abaixo do selo de nível:

```mdx
<VerifyNote lastVerified="2026-06-20" source="https://docs.anthropic.com/...">
One line on what's volatile here and to confirm at the source.
</VerifyNote>
```

- **`lastVerified`** = a data em que *você* realmente verificou contra a fonte.
- **`source`** = a página oficial que é a autoridade.

## Nunca fixe fatos de modelo no código

IDs/preços/contexto de modelos ficam **apenas** em `data/models.json`, renderizados por `<ModelTable />` na [página de modelos](/docs/whats-new/models-and-pricing). Na prosa, **faça link para lá** em vez de escrever um ID de modelo ou preço. Atualizar modelos = uma edição de uma linha nesse JSON (um primeiro PR perfeito).

## Respondendo a `needs-verification`

A automação (e os leitores) abrem issues rotuladas com `needs-verification` quando um fato pode estar desatualizado. Para resolver uma:

1. Verifique a afirmação contra a fonte oficial.
2. Se estiver correta → atualize `lastVerified` para hoje.
3. Se mudou → corrija o conteúdo **e** a data, e faça link para a fonte.
4. Se for um fato de modelo → atualize `data/models.json`, não a prosa.

## O princípio da honestidade

:::tip Uma data antiga é uma funcionalidade
Um `lastVerified` de meses atrás não significa que falhamos — ele diz ao leitor para checar novamente *aquele* fato. Melhor uma data antiga honesta do que uma errada com confiança. Não atualize uma data que você não verificou de fato.
:::
