Você ajustou um prompt e ele *parece* melhor — mas será que está? Sem **evals** (avaliações) você está voando às cegas: cada mudança é cara ou coroa, e você descobre que quebrou por causa de um usuário irritado, não de um teste. Evals transformam o "feeling" em um número em que você pode confiar, defender e acompanhar ao longo do tempo. Essa é a maior coisa que separa prompts de hobby do trabalho com Claude em nível de produção.

As ferramentas de eval e o comportamento dos modelos mudam rápido — reveja os guias indicados e **recalibre seu juiz sempre que mudar o modelo sob teste**. A OWASP ainda classifica errar isso (e prompt injection) entre os principais riscos de LLM para 2026.
</VerifyNote>

## A mentalidade: meça, não chute

Três regras que te salvam:

- **De baixo para cima vence de cima para baixo.** Colete falhas *reais* primeiro, depois projete a métrica para capturá-las. Um eval construído a partir de quebras reais prevê quebras reais; um eval inventado num quadro branco mede principalmente a sua imaginação.
- **Um número que você pode reexecutar.** Um eval é repetível: mesmas entradas → pontuação comparável. É isso que permite comparar o prompt v1 vs v2, ou `claude-haiku-4-5` vs `claude-sonnet-4-6`, com honestidade.
- **Barato de rodar, rode com frequência.** Se exigir uma tarde de um humano, não vai acontecer. Automatize.

## Construa um golden dataset (de baixo para cima)

Seu golden dataset é o coração de todo eval — um conjunto curado de entradas com expectativas reconhecidamente boas.

<Steps items={[
  {title: "Colete falhas reais primeiro", body: "Comece pelas saídas ruins de verdade: traces de produção, relatos de bugs, tickets de suporte. Esses são os casos que importam."},
  {title: "Ancore com 50–100 casos feitos à mão", body: "À mão, escreva casos cobrindo seus cenários mais críticos e mais propensos a erro. Esse é seu conjunto-âncora estável."},
  {title: "Expanda rumo a ~500+", body: "Adicione amostras de produção desidentificadas (remova PII) e casos sintéticos para cenários sub-representados. Não confie em métricas agregadas num conjunto minúsculo."},
  {title: "Faça-o crescer toda semana", body: "Cada nova regressão de produção vira um novo caso de teste. Um golden dataset é vivo, não congelado."},
]} />

## Pontuação: código primeiro, juiz depois

Recorra primeiro à verificação confiável mais barata.

- **Verificações programáticas (determinísticas)** — use sempre que a resposta tiver estrutura: correspondência exata/por palavra-chave, "JSON válido contra este schema", "chamou a ferramenta certa com os argumentos certos", "abaixo de N tokens / abaixo de X ms". Rápidas, gratuitas e nunca instáveis.
- **LLM-como-juiz** — para dimensões difusas (utilidade, tom, fidelidade a uma fonte) que resistem ao código. Dê ao juiz uma **rubrica**, não um feeling, e **calibre-o contra rótulos humanos** antes de confiar nele.

:::warning Juízes têm vieses
Juízes LLM tendem a favorecer respostas mais longas (viés de verbosidade) e qualquer opção mostrada primeiro (viés de posição). Defesas: uma rubrica rígida, comparação em pares em vez de pontuação absoluta, trocar a ordem das respostas e reavaliar o juiz contra uma fatia rotulada por humanos. Um juiz é **uma camada**, não o teste inteiro.
:::

<PromptCard title="Rubrica de LLM-como-juiz (inicial)">{`You are a strict grader. You are given a QUESTION, a REFERENCE answer, and a MODEL answer.
Score the MODEL answer from 1-5 on (a) faithfulness to the reference and (b) helpfulness.
Output ONLY JSON, nothing else: {"score": <1-5>, "reason": "<one short sentence>"}

QUESTION: {{question}}
REFERENCE: {{reference}}
MODEL: {{model_answer}}`}</PromptCard>

## Para agentes, teste também a *trajetória*

Um agente pode acertar a resposta final do jeito errado — entrando em loop, chamando uma ferramenta destrutiva ou queimando seu orçamento. Então avalie o caminho, não só o destino: **ele chamou as ferramentas certas, numa ordem sensata, sem loops, dentro do orçamento?** A correção das chamadas de ferramenta e as verificações de trajetória capturam falhas que um eval só do resultado final nunca enxerga.

## Ligue ao CI

É aqui que os evals dão retorno: tornar regressões impossíveis de mesclar.

<Steps items={[
  {title: "Rode o golden set em cada pull request", body: "Pontue programaticamente quando possível; rode o juiz no restante."},
  {title: "Falhe o build numa queda", body: "Defina um limiar (ex.: a pontuação não pode cair em relação à main). Uma mudança de prompt que regride a qualidade não pode subir."},
  {title: "Feche o ciclo em produção", body: "Quando o juiz sinaliza uma resposta ao vivo, encaminhe-a para uma fila de revisão humana; o revisor confirma, adiciona o caso ao golden set e reteste após a correção."},
]} />
