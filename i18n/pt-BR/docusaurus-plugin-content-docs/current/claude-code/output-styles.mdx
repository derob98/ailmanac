Os **estilos de saída** mudam *como* o Claude Code se comunica — seu papel, tom, verbosidade e se ele explica seu raciocínio — modificando o prompt do sistema diretamente. Eles mudam *como* o Claude responde, não *o que* ele sabe. Recorra a um quando você ficar reescrevendo o prompt pedindo a mesma voz ou formato a cada turno.

## Por que usá-los

- **Adeque-se ao seu fluxo de trabalho.** Conciso e orientado à ação quando você está avançando rápido; explicativo quando está aprendendo.
- **Modo de ensino.** Alguns estilos adicionam comentários educativos — ótimos enquanto você está conhecendo um codebase ou uma técnica.
- **Consistência.** Defina um estilo preferido uma vez para que cada sessão seja lida do jeito que você gosta.

## Os estilos integrados

O Claude Code vem com quatro:

| Estilo | O que faz |
|---|---|
| **Default** | O prompt de sistema padrão de engenharia de software — eficiente, orientado à ação. |
| **Proactive** | Executa imediatamente e faz suposições razoáveis em vez de pausar para decisões rotineiras. Orientação de execução autônoma mais forte do que o modo automático, mas você ainda vê solicitações de permissão antes que as ferramentas sejam executadas. |
| **Explanatory** | Adiciona "Insights" educativos entre as etapas, explicando escolhas de implementação e padrões do codebase. |
| **Learning** | Colaborativo, aprender-fazendo: compartilha Insights *e* pede que você escreva pequenos trechos por conta própria, deixando marcadores `[TODO(human)]` no seu código. |

Explanatory e Learning produzem respostas mais longas por design, então custam mais tokens de saída do que Default.

## Como trocar

Execute `/config` e escolha um estilo em **Output style**. Sua escolha é salva em `.claude/settings.local.json` no nível do projeto. Para defini-lo sem o menu, edite o campo `outputStyle` em qualquer arquivo de configurações:

```json
{
  "outputStyle": "Explanatory"
}
```

:::warning As mudanças exigem uma nova sessão
O estilo de saída faz parte do prompt do sistema, que o Claude Code lê **uma vez no início da sessão**. Uma mudança só tem efeito após `/clear` ou uma nova sessão. (O comando autônomo `/output-style` foi removido na v2.1.91 — use `/config` ou a configuração acima.)
:::

## Crie um estilo personalizado

Um estilo personalizado é um arquivo Markdown: frontmatter e, em seguida, as instruções a serem anexadas ao prompt do sistema. Salve-o no nível do usuário (`~/.claude/output-styles/`) ou no nível do projeto (`.claude/output-styles/`); o nome do arquivo se torna o nome do estilo, a menos que você defina `name`.

```markdown
---
name: Diagrams first
description: Lead every explanation with a diagram
keep-coding-instructions: true
---

When explaining code, architecture, or data flow, start with a Mermaid
diagram showing the structure, then explain in prose.
```

Defina `keep-coding-instructions: true` quando você estiver mudando apenas *como* o Claude se comunica, mas ainda quiser que ele programe normalmente. Deixe-o de fora (o padrão) quando o Claude não estiver fazendo engenharia de software de forma alguma — digamos, um assistente puramente de escrita ou de análise de dados — para que as instruções de programação integradas sejam descartadas.

## Estilos de saída vs CLAUDE.md vs outras alavancas

- **Estilo de saída** = uma predefinição de prompt do sistema para papel, tom e formato — aplica-se a *cada* resposta.
- **[CLAUDE.md](/docs/claude-code/claude-md)** = fatos e regras *do projeto* persistentes (convenções, salvaguardas), adicionados como uma mensagem após o prompt do sistema.
- **[Instruções personalizadas / estilos de resposta](/docs/claude-app/custom-instructions)** = o equivalente no app de consumo Claude.ai.

:::tip Use a alavanca certa
Quer que o Claude "sempre execute os testes antes de finalizar"? Isso é uma **salvaguarda do CLAUDE.md**, não um estilo. Quer que ele "explique cada etapa conforme avança"? Isso é um **estilo de saída**. Mantenha as regras de comportamento no CLAUDE.md e a apresentação nos estilos.
:::
