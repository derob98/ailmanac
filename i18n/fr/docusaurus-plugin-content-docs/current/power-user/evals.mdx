Vous avez ajusté un prompt et il *semble* meilleur — mais l'est-il vraiment ? Sans **evals** (évaluations), vous avancez à l'aveugle : chaque changement est un pile ou face, et vous découvrez qu'il a cassé quelque chose par un utilisateur mécontent, pas par un test. Les evals transforment le « feeling » en un chiffre auquel vous pouvez faire confiance, que vous pouvez défendre et surveiller dans le temps. C'est la seule chose qui sépare le plus nettement les prompts amateurs du travail Claude de qualité production.

## L'état d'esprit : mesurer, ne pas deviner

Trois règles qui vous sauvent :

- **Le bottom-up bat le top-down.** Collectez d'abord les *vrais* échecs, puis concevez la métrique pour les attraper. Une eval construite à partir de vraies pannes prédit les vraies pannes ; une eval inventée sur un tableau blanc mesure surtout votre imagination.
- **Un chiffre que vous pouvez réexécuter.** Une eval est reproductible : mêmes entrées → score comparable. C'est ce qui vous permet de comparer honnêtement le prompt v1 vs v2, ou `claude-haiku-4-5` vs `claude-sonnet-5`.
- **Peu coûteuse à exécuter, exécutée souvent.** Si cela prend un après-midi à un humain, cela n'arrivera pas. Automatisez-la.

## Construire un golden dataset (bottom-up)

Votre golden dataset est le cœur de chaque eval — un ensemble organisé d'entrées avec des attentes connues comme bonnes.

<Steps items={[
  {title: "Collectez d'abord les vrais échecs", body: "Partez des mauvaises sorties réelles : traces de production, rapports de bugs, tickets de support. Ce sont les cas qui comptent."},
  {title: "Ancrez avec 50 à 100 cas façonnés à la main", body: "À la main, rédigez des cas couvrant vos scénarios les plus critiques et les plus sujets aux erreurs. C'est votre ensemble d'ancrage stable."},
  {title: "Étendez vers ~500 et plus", body: "Ajoutez des échantillons de production désidentifiés (retirez les PII) et des cas synthétiques pour les scénarios sous-représentés. Ne faites pas confiance aux métriques agrégées sur un petit ensemble."},
  {title: "Faites-le grandir chaque semaine", body: "Chaque nouvelle régression de production devient un nouveau cas de test. Un golden dataset est vivant, pas figé."},
]} />

## Noter : le code d'abord, le juge ensuite

Optez d'abord pour la vérification fiable la moins coûteuse.

- **Vérifications programmatiques (déterministes)** — utilisez-les partout où la réponse a une structure : correspondance exacte/par mot-clé, « JSON valide selon ce schéma », « a-t-il appelé le bon outil avec les bons arguments », « sous N tokens / sous X ms ». Rapides, gratuites et jamais capricieuses.
- **LLM-as-judge** — pour les dimensions floues (utilité, ton, fidélité à une source) qui résistent au code. Donnez au juge une **grille (rubric)**, pas un feeling, et **calibrez-le par rapport à des labels humains** avant de lui faire confiance.

:::warning Les juges ont des biais
Les juges LLM dérivent vers les réponses plus longues (biais de verbosité) et vers l'option affichée en premier (biais de position). Défenses : une grille stricte, la comparaison par paires au lieu d'une notation absolue, l'inversion de l'ordre des réponses, et la revérification du juge par rapport à une tranche labellisée par des humains. Un juge est **une couche**, pas le test complet.
:::

Score the MODEL answer from 1-5 on (a) faithfulness to the reference and (b) helpfulness.
Output ONLY JSON, nothing else: {"score": <1-5>, "reason": "<one short sentence>"}

QUESTION: {{question}}
REFERENCE: {{reference}}
MODEL: {{model_answer}}`}</PromptCard>

## Pour les agents, testez aussi la *trajectoire*

Un agent peut atterrir sur la bonne réponse finale de la mauvaise manière — en bouclant, en appelant un outil destructeur, ou en épuisant votre budget. Alors évaluez le chemin, pas seulement la destination : **a-t-il appelé les bons outils, dans un ordre sensé, sans boucles, dans le budget ?** La justesse des appels d'outils et les vérifications de trajectoire attrapent des échecs qu'une eval portant uniquement sur la réponse finale ne voit jamais.

## Câblez-la dans la CI

C'est là que les evals paient : rendre les régressions impossibles à merger.

<Steps items={[
  {title: "Exécutez le golden set sur chaque pull request", body: "Notez de façon programmatique quand c'est possible ; exécutez le juge sur le reste."},
  {title: "Faites échouer le build en cas de baisse", body: "Fixez un seuil (par ex. le score ne doit pas chuter par rapport à main). Un changement de prompt qui dégrade la qualité ne peut pas être livré."},
  {title: "Bouclez la boucle en production", body: "Quand un juge signale une réponse en direct, routez-la vers une file de revue humaine ; le relecteur confirme, ajoute le cas au golden set et retteste après le correctif."},
]} />
