Bonne nouvelle : **les fondamentaux du prompting se transfèrent à tous les modèles.** Si vous savez bien prompter Claude, vous savez bien prompter ChatGPT, Gemini ou Llama. Quelques détails de surface diffèrent — voici la pierre de Rosette.

## Ce qui se transfère partout

Ces éléments fonctionnent sur tous les modèles — ils relèvent de la *communication*, pas d'un fournisseur :

- **Donner du contexte** : objectif, audience, format, ton ([Les bases du prompting](/docs/prompting/basics)).
- **Des exemples** ([few-shot](/docs/prompting/few-shot)).
- **Demander un raisonnement** sur les tâches difficiles.
- **Contraindre la sortie** (longueur, format, JSON uniquement).
- **Itérer** en langage naturel.
- **Rôles / prompts système** — chaque grand modèle possède ce concept ([Rôles](/docs/foundations/roles)).

Apprenez-les une fois et vous êtes portable.

## Ce qui diffère (et ce qu'il faut ajuster)

| Aspect | À noter lors d'un changement |
|---|---|
| **Repères de structure** | Claude adore les [balises XML](/docs/prompting/xml-tags) ; les autres modèles les acceptent mais peuvent préférer les titres Markdown ou le JSON. |
| **Intensité du ton** | Les anciens modèles toléraient les prompts insistants ou EN MAJUSCULES ; les modèles récents, plus littéraux, préfèrent des instructions calmes et précises. Réajustez selon le modèle. |
| **Raisonnement** | Penser étape par étape aide certains ; les modèles de raisonnement dédiés le font en interne — n'en rajoutez pas. |
| **Préremplissage** | Mettre des mots dans la bouche de l'assistant est un levier propre à Claude ; le support varie ailleurs. |
| **Poids du prompt système** | Tous en ont un, mais la force avec laquelle il est suivi varie — testez. |
| **Appel d'outils / de fonctions** | Même concept, schémas de requête différents selon le fournisseur. |

## Un workflow portable

<Steps items={[
  {title: "Écrire avec une structure universelle", body: "Rédigez le prompt avec contexte + format + exemples — les parties qui se transfèrent à n'importe quel modèle. N'intégrez pas encore de syntaxe propre à un fournisseur."},
  {title: "Tester sur votre modèle cible", body: "Faites passer quelques entrées réelles dans le modèle sur lequel vous allez réellement déployer (/docs/foundations/evals). Jugez les sorties, pas votre intuition sur le modèle."},
  {title: "Ajuster uniquement les détails de surface", body: "Si quelque chose cloche, changez un seul levier de surface à la fois — style de délimiteur, intensité du ton, demande de raisonnement — pas tout le prompt."},
  {title: "Garder les prompts sous contrôle de version", body: "Stockez les prompts sous forme de fichiers pour que changer de modèle soit un petit diff révisable plutôt qu'une réécriture."}
]} />

Comme seule la *surface* change, un prompt bien structuré se transfère en général avec un ajustement d'une seule ligne. En voici un écrit de façon universelle — contexte, format explicite et tâche claire, sans syntaxe propre à un fournisseur à retirer :

<PromptCard title="Un prompt portable et neutre">{`You are a technical editor. Rewrite the text below to be clearer and more concise.

Constraints:
- Keep every fact unchanged.
- Target a general professional audience.
- Return only the rewritten text, no preamble.

Text:
"""
[paste the draft here]
"""`}</PromptCard>

Ce prompt s'exécute tel quel sur Claude, GPT, Gemini ou un modèle ouvert. Si un modèle ignore la consigne de ne renvoyer que le texte réécrit, c'est votre unique ajustement de surface — resserrez la contrainte ou, sur Claude, [préremplissez](/docs/prompting/prompting-claude) la réponse.

:::tip Ne réécrivez pas de zéro
Passer à un nouveau modèle exige rarement un nouveau prompt — le plus souvent juste un ajustement de ton ou de format. Votre structure et vos exemples se reportent.
:::
