**ローカルAIエージェント**とは、自分のハードウェア上で完全に動作する自律ループのことです。オープンウェイトモデル（[Ollama](/docs/models/run-models-locally-ollama) や LM Studio が提供）が何をすべきかを判断し、あなたが与えた**ツール**を呼び出し、その結果を読み取り、タスクが完了するまで動き続けます — しかも**何ひとつマシンの外に出ません**。クラウドAPIなし、呼び出しごとの請求なし、インターネット不要。ただし注意点があります。ラップトップで動かせるほど小さいモデルは、難しい推論や長期的な計画立案においてフロンティアモデルより劣り、その信頼性と安全性は*あなた自身*が担うことになります。このページでは、ローカルエージェントを正直に評価した上での採用理由、最小限のアーキテクチャ、実際にローカルで動くもの、そして最初の1台を作るための現実的な道筋を扱います。

## なぜ*ローカル*エージェントを構築するのか（そしてしないべきとき）

通常の[ツール利用エージェント](/docs/api/tool-use)はクラウドモデルを呼び出します。*ローカル*エージェントは、そのクラウド呼び出しを自分のマシン上で動くモデルに置き換えます。ある程度の能力を手放し、いくらかの運用負担を引き受ける代わりに、他の方法では得がたい4つのものが手に入ります。

- **プライバシー** — プロンプト、ツール入力、ツール出力がマシンの外に出ることは一切ありません。規制対象、機密性が高い、あるいはエアギャップ環境のチームがローカルエージェントを構築する理由はまさにこれです。データが物理的に第三者へ渡りようがないのです。
- **オフライン** — インターネット不要、API依存なし、プロバイダーの障害なし。エージェントはディスク上のファイルであり、飛行機の中でもファイアウォールの背後でも動きます。
- **呼び出しごとのコストなし** — エージェントループは1タスクあたり数十回のモデル呼び出しを行うことがあります。ローカルではそれらの呼び出しは「無料」です（トークンではなく電気代とハードウェアで支払います）。なのでメーターを気にせず反復させられます。
- **完全な制御** — 正確なモデルバージョンを固定し、挙動をカスタマイズし、レート制限や利用規約の突然の変更なしに動かせます。

正直なトレードオフ — コミットする前にこれらを冷静に見極めてください。

- **能力ギャップ。** エージェントで最も難しいのは*推論*です。多段階の作業を計画し、失敗したツール呼び出しから回復し、いつ止めるべきかを知ること。ラップトップで動かせるモデル（おおよそ1B〜14Bパラメータ）は、この点でフロンティアモデルより明らかに劣ります。シンプルで範囲の狭いループはローカルでもよく機能しますが、長期的でオープンエンドなタスクこそ、ローカルエージェントが最も脱線しやすい領域です。
- **信頼性と安全性はあなたの責任。** あなたのためにフィルタリング、監視、ガードレール設定をしてくれるプロバイダーはいません。エージェントが永遠にループしたり、間違ったツールを呼んだり、破壊的なアクションを取ったりしたら、それはあなたの設計の責任です。（下の警告を参照 — ここは人々が過小評価する部分です。）
- **ハードウェアの制約。** より大きく賢いモデルは、ほとんどのマシンが持つよりも多くのRAM/VRAMを必要とします。たいていは存在する中で最良のモデルではなく、*自分のハードウェアで動かせる中で最も能力の高いモデル*を選ぶことになります。

確かな経験則: **ローカルで始め、タスクが要求したときにエスカレーションする。** プライベート/オフライン/大規模でも安価な作業や、範囲の狭いループにはローカルエージェントを使い、タスクが本当に追加の推論を必要とするときにはフロンティアAPIエージェントに手を伸ばします。下のアーキテクチャはどちらでも同一です — 変わるのはエンドポイントだけ — なので、ローカルでプロトタイプを作り、後でモデルを差し替えられます。

## 最小限のアーキテクチャ

エージェントを核まで削ぎ落とすと、4つの部品があります。それ以外のすべては、これらの上に乗った便利機能です。

```
        ┌─────────────────────────────────────────────┐
        │                                             │
        │   1. LOCAL MODEL  ──►  decides next action  │
        │      (Ollama / LM Studio, tool-capable)     │
        │                  │                          │
        │                  ▼                          │
        │   2. TOOL-CALLING LOOP                       │
        │      parse the model's tool request,         │
        │      run it, feed the result back            │
        │                  │                          │
        │                  ▼                          │
        │   3. TOOLS  ──►  search / read file /        │
        │      run code / call an API (your code)      │
        │                  │                          │
        │                  ▼                          │
        │   4. GUARDRAIL / STOP CONDITION              │
        │      max steps, budget, approval gate,       │
        │      "done" check ──► exit the loop          │
        │                                             │
        └─────────────────────────────────────────────┘
```

1. **ツール呼び出しに対応したローカルモデル。** モデルは、単にチャットするだけでなく、ツールを呼び出すための構造化されたリクエスト（別名*関数呼び出し*）を出力できる必要があります。Ollama はこれを自身のAPIと、`http://localhost:11434/v1` にある**OpenAI互換エンドポイント**の両方を通じて公開しているため、OpenAI形式を話すフレームワークならどれでもローカルモデルを動かせます。<VerifyNote lastVerified="2026-06-28" source="https://ollama.com/blog/tool-support">Ollama はツール呼び出しに対応しており、Tools カテゴリの下にツール対応モデルを掲載しています。OpenAI互換エンドポイント（`/v1`）もツールに対応しています。どのモデルが正確にツールに対応するかは新しいリリースが出るたびに変わります — 最新の Tools 一覧を <a href="https://ollama.com/search?c=tools">ollama.com/search?c=tools</a> で確認してください。</VerifyNote>
2. **ツール呼び出しループ。** エージェントの心臓部です。会話をモデルに送り、ツールを呼び出すよう求めてきたか確認し、そのツールを実行し、結果を追加し、繰り返します。モデルがツールを要求せずに回答したら、ループは終了します。
3. **ツール。** モデルに公開する単なる関数です — Web検索、ファイル読み取り、シェルコマンド実行、データベースへのクエリ、APIの呼び出し。各ツールには名前、説明、型付きの入力スキーマがあり、モデルはいつどう使えばよいかを知ります。
4. **ガードレール/停止条件。** 自律性には必須です。最低限、ループが永遠に走れないようにする**最大ステップ上限**、そして — 書き込み、削除、支払い、送信を行うものすべてには — **承認ゲート**かサンドボックスを設けます。これがなければ、それはエージェントではなく、ファイルアクセス権を持った無限ループです。

ステップ2のループは本当に小さなものです。ローカルの Ollama エンドポイントに対する Python の擬似コードで示すと、こうなります。

<PromptCard title="最小限のローカルエージェントループ（Python擬似コード、ローカルOllamaを指す）">{`from openai import OpenAI

# Point the OpenAI client at your LOCAL Ollama endpoint — nothing leaves the machine
client = OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")

tools = [{
    "type": "function",
    "function": {
        "name": "read_file",
        "description": "Read a UTF-8 text file and return its contents",
        "parameters": {
            "type": "object",
            "properties": {"path": {"type": "string"}},
            "required": ["path"],
        },
    },
}]

def run_tool(name, args):
    if name == "read_file":
        # GUARDRAIL: only allow reads inside a sandboxed directory
        return safe_read(args["path"])
    raise ValueError(f"unknown tool: {name}")

messages = [{"role": "user", "content": "Summarize ./notes/today.md"}]

for step in range(8):                       # GUARDRAIL: hard step cap
    resp = client.chat.completions.create(
        model="llama3.1", messages=messages, tools=tools,
    )
    msg = resp.choices[0].message
    messages.append(msg)

    if not msg.tool_calls:                  # STOP: model answered, we're done
        print(msg.content)
        break

    for call in msg.tool_calls:
        result = run_tool(call.function.name, json.loads(call.function.arguments))
        messages.append({
            "role": "tool", "tool_call_id": call.id, "content": str(result),
        })
else:
    print("Stopped: hit the step cap without finishing.")`}</PromptCard>

これがパターンのすべてです。フレームワークはこの上にメモリ、リトライ、マルチエージェントのオーケストレーション、トレーシング、構造化された状態を加えます — しかしそのどれもが、このループのより堅牢なバージョンにすぎません。

## どのローカルモデルがエージェント的/ツール利用の作業に向くか

すべてのオープンウェイトモデルがエージェントを動かせるわけではありません。基準は**信頼できるツール呼び出し**です。モデルは整形式のツールリクエストを一貫して出力し、正しいツールを選び、引数を幻覚しないことが求められます。選ぶ際の2つのフィルター。

- **ツール対応モデルでなければならない。** Ollama はこれらにタグを付けています — 思い込みではなく、[Tools カテゴリ](https://ollama.com/search?c=tools)で最新の一覧を見てください。確実なローカルツール利用でよく挙げられるモデルには Qwen と Llama の指示チューニング済みファミリーがありますが、最良の選択は四半期ごとに動きます。
- **コンテキストの余裕を持ってハードウェアに収まらなければならない。** エージェントループは長いメッセージ履歴を蓄積します（すべてのツール結果が追加されます）。なので重みと*潤沢なコンテキストウィンドウ*の両方がメモリに収まる必要があります。余裕を持って収まり高速に動く小さめのモデルは、ディスクにスワップしてループの途中で失速する大きめのモデルにしばしば勝ります。

<VerifyNote lastVerified="2026-06-28" source="https://ollama.com/search?c=tools">
具体的なモデル名とそのツール呼び出しの信頼性は、新しいオープンウェイトのリリースが出るたびに速く変わります。名前の挙がったモデル（Llama、Qwen、Mistral など）はすべて生鮮品として扱ってください — Ollama の最新の Tools カテゴリと、コミット前の自前の簡単な評価に照らして確認しましょう。オープンモデルの全体像をより詳しく知るには <a href="/docs/models/deepseek-qwen-open-models">DeepSeek と Qwen のオープンモデル</a>を参照してください。
</VerifyNote>

決め手となる一手はベンチマークを読むことではありません — *自分の*タスクの小さな[評価（eval）](/docs/power-user/evals)を、候補となる2〜3個のモデルに対して実行することです。リーダーボードのトップを取るモデルでも、エージェントが必要とする特定のツールでは信頼できないことがあります。自分のループ上で測定しましょう。

## ローカルで動作するフレームワーク

上のループを自前で組むこともでき、最初のエージェントにはそれが学ぶうえで素晴らしい方法です。実用的なものになると、フレームワークがリトライ、メモリ、マルチエージェント調整、トレーシングを提供してくれます。重要な事実: 人気のエージェントフレームワークは**モデル非依存**です — モデルがクラウドにあるか `localhost` にあるかを気にせず、正しいエンドポイントを指してさえいればよいのです。

- **[LangGraph](https://github.com/langchain-ai/langgraph)** — ステートフルなエージェントのための低レベルなオーケストレーションフレームワーク（永続的な実行、永続化、ヒューマン・イン・ザ・ループ）。モデル非依存で、LangChain の Ollama 連携を介して回避策なしにローカルモデルへ接続できます。エージェントの状態グラフを明示的に制御する必要があるときに適しています。
- **[CrewAI](https://docs.crewai.com/en/learn/llm-connections)** — 1つ以上のロールベースのエージェント（「クルー」）をオーケストレーションする高レベルなフレームワーク。LiteLLM を介してモデル非依存で、`LLM(model="ollama/llama3.1", base_url="http://localhost:11434")` でエージェントをローカルモデルに向けられます。複数の協調するエージェントを素早く組み立てたいときに適しています。
- **[OpenAI Agents SDK](https://openai.github.io/openai-agents-python/)** — 軽量なマルチエージェントフレームワーク。名前に反して**プロバイダー非依存**です。[LiteLLM 連携](https://openai.github.io/openai-agents-python/models/litellm/)を介して、OpenAI のモデルではなくローカルの Ollama モデルへ向けられます。OpenAI のエージェントの使い心地をローカルバックエンドで得たいときに適しています。

<VerifyNote lastVerified="2026-06-28" source="https://openai.github.io/openai-agents-python/models/litellm/">
3つのフレームワークはいずれも検証日時点でローカルのオープンウェイトモデルを動かせました — LangGraph と CrewAI はネイティブに、OpenAI Agents SDK は LiteLLM アダプターを介して。アダプターの対応状況、インストール用の extras、正確な設定（モデル名のプレフィックス、ベースURL）はリリース間で変わります。構築する前に各プロジェクトの最新ドキュメントに照らして確認してください。これらおよび他のフレームワークのより広い比較については <a href="/docs/models/open-source-agent-frameworks">オープンソースのエージェントフレームワーク</a>を参照してください。
</VerifyNote>

3つすべてをつまみ食いするのではなく、1つのフレームワークを選んでよく学びましょう。概念（エージェント、ツール、ループ、状態）は転用できます。APIは細部にすぎません。

## 最初のローカルエージェントを構築する

現実的な道筋は、「ループがまったくない」状態から「ガード付き自律ループ」へ、意図的なステップで進みます。ステップ4に飛ばさないでください — ローカルエージェントで人々がぶつかる失敗のほとんどは、弱いモデルに早すぎる段階で自由を与えすぎることから来ます。

<Steps items={[
  {title: "ツール対応モデルをローカルで動かす", body: "Ollama をインストールし（「ローカルでモデルを動かす」を参照）、ツール用にタグ付けされたモデルをプルします。例: ollama pull llama3.1。http://localhost:11434 で提供され、ollama list に表示されることを確認します。まだエージェントはありません — ただ呼び出せるモデルがあるだけです。"},
  {title: "ツール呼び出しを1回だけ動かす（ループなし）", body: "1つのツールを定義した（例: get_time や read_file 関数）単一のリクエストを送り、モデルが本当に有効な引数を持つ整形式のツール呼び出しを返すか確認します。モデルがきれいなツール呼び出しを1回でも確実にできないなら、ループでは生き残れません — 後ではなく今、モデルを切り替えましょう。"},
  {title: "それを境界付きループで包む", body: "上の PromptCard の「実行→結果を返す」ループを、ハードな最大ステップ上限（6〜8から開始）とともに追加します。読み取り専用ツールで、小さく範囲の狭いタスクを与えます。すべてのステップが出力されるのを見て、モデルの推論を確認し、ループしたりツールを誤用したりするのを捕まえられるようにします。"},
  {title: "書き込み/破壊的ツールの前にガードレールを追加する", body: "ここで初めて、状態を変えるツール（ファイルを書く、コマンドを実行する、お金のかかるAPIを呼ぶ）を導入します。それぞれを承認プロンプトの背後に置くか、サンドボックス/コンテナで実行し、予算または実時間の上限を加えます。失敗モードを意図的にテストします: 完了できないタスクを与えて、きれいに停止することを確認してください。"},
  {title: "フレームワークへ卒業する（任意）", body: "自前のループが動いたら、ローカルエンドポイントを指す LangGraph、CrewAI、または OpenAI Agents SDK に移植します。リトライ、メモリ、マルチエージェントのオーケストレーションが無料で手に入り — モデルはあなたのマシン上の、まさにその場所にとどまります。"},
]} />
