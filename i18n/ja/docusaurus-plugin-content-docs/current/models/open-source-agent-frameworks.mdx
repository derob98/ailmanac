エージェントをいくつか手作業で構築すると、同じ配管作業が何度も現れます。モデルを呼び出し、要求されたツールを実行し、結果を返し、仕事が完了したら停止するループです。**エージェントフレームワーク**はその配管 — さらに状態、メモリ、マルチエージェント協調 — をパッケージ化してくれるので、書くグルーコードが減ります。このページは、主要な**オープンソース**の選択肢、それらが実際に何を提供するのか、そしてどう選ぶのかを、プロバイダーに依存しない形で長く使える地図として示します。それらのほとんどは**モデル非依存**です。Claude、GPT、Gemini、そしてローカルのオープンウェイトモデルで動作します。

## エージェントフレームワークが実際に与えてくれるもの

ブランディングを取り払うと、フレームワークが提供しているのは次の4つのうちのいくつかです：

- **オーケストレーション** — 制御ループ。逐次ステップ、分岐、リトライ、完了までのループ、そして（ますます増えている）クラッシュを生き延びて停止した場所から再開する**永続的実行（durable execution）**。
- **ツール** — モデルが呼び出せる関数を宣言し、その引数を検証し、実行し、結果を返すための標準的な方法。あなたが[ツール使用](/docs/api/tool-use)ですでに知っている、宣言→呼び出し→実行→返却 と同じループです。
- **メモリと状態** — 会話履歴、スクラッチパッドの事実、取得したドキュメントをターンをまたいで保持する場所。これにより、エージェントはステップ間で記憶喪失にならずに済みます。
- **マルチエージェント** — 複数の専門化されたエージェントが互いに仕事を渡し合うパターン。ワーカーに委譲するスーパーバイザー、あるいはタスクで協働する対等なエージェントなど。（概念的にはClaude Codeの[サブエージェント](/docs/claude-code/subagents)と同じ発想です。）

フレームワークが価値を持つのは、それがなければこの4つすべてを自分で書いて保守することになる場合です。タスクが「モデルを呼び出し、1つか2つのツールを実行し、答えを返す」だけなら価値は*ありません* — その場合、フレームワークはあなたが格闘して時間を費やすオーバーヘッドになります。

## 3つの原型

フレームワークは、そのマーケティングが示唆するほどには違いません。実際には3つの形があり、ほとんどのプロジェクトはそのうちの1つのバリエーションです：

- **ステートフルなグラフ / ワークフロー。** エージェントをノードとエッジ（あるいはステップと遷移）からなる明示的なグラフとしてモデル化します。最大限の制御、検査可能な状態、長時間実行や人間がループに介在するフローに適しています。最初に学ぶことは多めです。→ **LangGraph**、**LlamaIndex Workflows**。
- **ロールベースのクルー。** エージェントを*役割*（「リサーチャー」「ライター」「レビュアー」）で記述し、協働させたりプロセスを実行させたりします。マルチエージェントのチームを素早く表現できますが、高レベルの抽象化と引き換えに、きめ細かい制御の一部を手放します。→ **CrewAI**、そして**AutoGen**の会話型マルチエージェントスタイル。
- **最小ループ / 抽象化が少ない。** モデルのネイティブなツール呼び出しの上に薄いレイヤーを乗せ、少数のエージェント間のハンドオフを加えただけで、それ以外はほとんどありません。端から端まで読みやすく、捨てやすい。→ **OpenAI Agents SDK**（実験的な**Swarm**の本番向け後継）、そして**あなた自身が書く素のループ**。

## 簡単なツアー（コミットする前にポジショニングを確認）

これらは知っておく価値のあるオープンプロジェクトです。名前を挙げているのは、それぞれの実際のリポジトリを検証したからです。移ろいやすいものはすべて上記のVerifyNoteの裏側にあります。

- **LangGraph** — グラフとしてモデル化された、ステートフルで長時間実行のエージェントのための低レベルなオーケストレーションフレームワーク。永続的実行と人間がループに介在することがファーストクラスです。単体でも、より広い**LangChain**エコシステムと組み合わせても使えます。モデル非依存。<VerifyNote lastVerified="2026-06-28" source="https://github.com/langchain-ai/langgraph" />
- **LlamaIndex** — データ/RAGフレームワーク（コネクタ、インデックス、検索）として始まり、現在はエージェント向けのイベント駆動な**Workflows**レイヤーも提供しています。エージェントが本質的に*あなたのドキュメントに対する検索*である場合に強力です。<VerifyNote lastVerified="2026-06-28" source="https://github.com/run-llama/llama_index" />
- **Microsoft AutoGen** — 会話型マルチエージェントシステムのためのフレームワーク。2026年中頃の時点で**メンテナンスモード**にあります。Microsoftは新規プロジェクトを統一された後継（**Microsoft Agent Framework**、AutoGen + Semantic Kernelを統合）へ誘導しています。開始前に現在のステータスを確認してください。<VerifyNote lastVerified="2026-06-28" source="https://github.com/microsoft/autogen" />
- **CrewAI** — ロールベースのフレームワーク：エージェントを役割と目標で定義し、**Crews**（自律的な協働）または**Flows**（イベント駆動の制御）に編成します。マルチエージェントチームへの近道です。<VerifyNote lastVerified="2026-06-28" source="https://github.com/crewAIInc/crewAI" />
- **OpenAI Agents SDK** — ハンドオフを伴うマルチエージェントワークフローのための、意図的に軽量で抽象化の少ないフレームワーク。名前に反して**プロバイダー非依存**であり（ドキュメントは100以上のLLMのサポートを記載）、実験的な**Swarm**の本番対応の後継です。<VerifyNote lastVerified="2026-06-28" source="https://github.com/openai/openai-agents-python" />
- **素のエージェントループ** — フレームワークはまったくなし：モデルのネイティブなツール呼び出しを囲む、あなた自身の`while`ループ。シンプルなエージェントにとって正しいデフォルトであり、上記のあらゆるフレームワークが結局のところラップしているものです。

## あらゆるフレームワークがラップしているもの

どんなライブラリに手を伸ばす前にも、それらすべてが構築されている上のループを見ておくと役立ちます。これが発想のすべてです — モデルが決め、あなたが実行し、完了するまで繰り返す：

```python
# Provider-neutral agent loop — the core every framework wraps.
# `model_call` and `run_tool` are yours; swap in Claude, GPT, Gemini, or a local model.

def agent_loop(task, tools, max_steps=10):
    messages = [{"role": "user", "content": task}]

    for _ in range(max_steps):
        # 1. Ask the model what to do next (it sees the tool schemas).
        response = model_call(messages, tools=tools)

        # 2. No tool requested → the model is done. Return its answer.
        if not response.tool_calls:
            return response.text

        # 3. Run each requested tool and feed results back in.
        messages.append(response.as_message())
        for call in response.tool_calls:
            result = run_tool(call.name, call.arguments)
            messages.append({
                "role": "tool",
                "tool_call_id": call.id,
                "content": result,
            })

    return "Stopped: hit max_steps without finishing."
```

これが読めるなら、このページのあらゆるフレームワークが内部で何をしているかが分かります。グラフは明示的な状態と分岐を加え、クルーは役割と委譲を加え、最小SDKは整然としたハンドオフを加えます — しかし鼓動は常にこのループです。

## 誇大宣伝についての注意

ここにあるどのフレームワークも「最高」ではありません — その問い自体が誤った形をしています。グラフ系フレームワークは制御と永続性で勝ち、クルー系フレームワークはチームを素早く表現することで勝ち、最小系は読みやすさと低いロックインで勝ち、そして素のループはフレームワークのREADMEが認めるよりも頻繁に勝ちます。正しい選択は、*あなたの*最も難しいステップを明確にする、最も小さなツールです。モデルを選ぶときと同様に、スター数ではなく、あなた自身のタスクに決めさせましょう。[評価（evals）](/docs/power-user/evals)で適用するのと同じ規律がここにも当てはまります：プロトタイプを作り、実際のケースで測定し、出口ランプを残しておく。
