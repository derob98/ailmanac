良い知らせがあります。**プロンプティングの基本原則は、どのモデルにも引き継げます。** Claudeを上手にプロンプトできるなら、ChatGPT、Gemini、Llamaも上手にプロンプトできます。表面的な細部はいくつか異なります — これがそのロゼッタストーンです。

## どこでも引き継げるもの

これらはどのモデルでも機能します — ベンダーではなく*コミュニケーション*に関わるものだからです。

- **背景を与える**: 目的、読み手、形式、トーン（[プロンプティングの基礎](/docs/prompting/basics)）。
- **例**（[few-shot](/docs/prompting/few-shot)）。
- 難しいタスクでは**推論を求める**。
- **出力を制約する**（長さ、形式、「JSONのみ」）。
- 平易な言葉で**反復する**。
- **役割 / システムプロンプト** — 主要なモデルにはすべてこの概念があります（[役割](/docs/foundations/roles)）。

一度これらを身につければ、どこにでも持ち運べます。

## 異なるもの（と調整すべきこと）

| 観点 | 切り替え時の注意点 |
|---|---|
| **構造の手がかり** | Claudeは[XMLタグ](/docs/prompting/xml-tags)を好みます。他のモデルも受け付けますが、Markdownの見出しやJSONを好む場合があります。 |
| **トーンの強さ** | 古いモデルは強い口調やALL-CAPSのプロンプトを許容しました。指示に忠実な新しいモデルは、落ち着いた具体的な指示を好みます。モデルごとに調整し直しましょう。 |
| **推論** | 「ステップごとに考えて」は一部のモデルで役立ちます。専用の推論モデルは内部で行うので — 重ねて指示しないでください。 |
| **プレフィル** | アシスタントの発言の冒頭に言葉を入れるのはClaude流のレバーです。他では対応状況がまちまちです。 |
| **システムプロンプトの重み** | すべてに存在しますが、どれだけ強く従われるかは異なります — テストしましょう。 |
| **ツール / 関数呼び出し** | 概念は同じですが、リクエストのスキーマはプロバイダーごとに異なります。 |

## 持ち運べるワークフロー

<Steps items={[
  {title: "普遍的な構造で書く", body: "背景 + 形式 + 例で、つまりどのモデルにも引き継げる部分でプロンプトを起草します。まだベンダー固有の構文を組み込まないでください。"},

  {title: "表面的な細部だけを調整する", body: "何かがおかしければ、プロンプト全体ではなく、表面のレバーを一度に一つずつ変えます — 区切り記号のスタイル、トーンの強さ、推論を求めるかどうか。"},
  {title: "プロンプトをバージョン管理に置く", body: "プロンプトをファイルとして保存すれば、モデルの切り替えは書き直しではなく、小さくレビュー可能な差分になります。"}
]} />

*表面*だけが変わるので、うまく構造化された一つのプロンプトはたいてい一行の微調整で移植できます。以下は普遍的なやり方で書かれた一例です — 背景、明示的な形式、明確なタスクで、剥ぎ取るべきベンダー固有の構文はありません。

<PromptCard title="持ち運べる、ベンダー中立なプロンプト">{`You are a technical editor. Rewrite the text below to be clearer and more concise.

Constraints:
- Keep every fact unchanged.
- Target a general professional audience.
- Return only the rewritten text, no preamble.

Text:
"""
[paste the draft here]
"""`}</PromptCard>

そのプロンプトはClaude、GPT、Gemini、あるいはオープンモデルでもそのまま動きます。もしあるモデルが「書き直した文章だけを返す」を無視するなら、それがあなたの唯一の表面的な微調整です — 制約を締めるか、Claudeなら返信を[プレフィル](/docs/prompting/prompting-claude)しましょう。

:::tip ゼロから書き直さないこと
新しいモデルへの移行で、新しいプロンプトが必要になることはまれです — たいていはトーンや形式の微調整だけです。構造と例はそのまま引き継がれます。
:::
