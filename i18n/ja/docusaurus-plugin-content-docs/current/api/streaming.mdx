API 上でチャットのような体験を構築する際の、2 つの実務的な現実: ユーザーが出力をすぐ見られるよう**ストリーミング**すること、そして API がステートレスなので**履歴を自分で管理**することです。

## ストリーミング

ストリーミングがないと、ユーザーは返信全体を待ちます。ストリーミングがあれば、トークンは生成されるそばから届きます — 体感速度がはるかに良くなります。SDK のストリーミングヘルパーを使いましょう:

<Tabs groupId="lang">
<TabItem value="python" label="Python">

```python
with client.messages.stream(
    model="claude-sonnet-4-6", max_tokens=1024,
    messages=[{"role": "user", "content": "Explain RAG in two sentences."}],
) as stream:
    for text in stream.text_stream:
        print(text, end="", flush=True)
```

</TabItem>
<TabItem value="ts" label="TypeScript">

```ts
const stream = client.messages.stream({
  model: "claude-sonnet-4-6", max_tokens: 1024,
  messages: [{ role: "user", content: "Explain RAG in two sentences." }],
});
for await (const event of stream) {
  if (event.type === "content_block_delta") process.stdout.write(event.delta.text ?? "");
}
```

</TabItem>
</Tabs>

## マルチターン: 履歴はあなたが保持する

API は呼び出しの間に**メモリを持ちません**（[理由](/docs/foundations/tokens-and-context)）。会話を継続するには、毎回**それまでのやり取り全体**を送り返します:

```python
messages = [{"role": "user", "content": "Hi, I'm planning a trip."}]
# ... get assistant reply, then append both turns:
messages.append({"role": "assistant", "content": assistant_text})
messages.append({"role": "user", "content": "Make it 3 days."})
# send the full `messages` list again
```

## 長い会話はウィンドウを埋めていく

履歴が増えると[コンテキストウィンドウ](/docs/foundations/tokens-and-context)を消費し、コストも上がります。対策:

- 古いターンを、引き継いでいく短いまとめへと**要約/圧縮**する。
- 関係のない以前のターンを**トリミング**する。
- **[プロンプトキャッシュ](/docs/api/prompt-caching)** と組み合わせて、安定した先頭部分への再支払いを避ける。
