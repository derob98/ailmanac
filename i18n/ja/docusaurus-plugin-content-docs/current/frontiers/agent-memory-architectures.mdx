# エージェントのメモリは実際にどう動くのか

チャットボットに同じ質問を別々のセッションで二度すると、どちらのときも初対面のように答える。これはバグではない — デフォルトの挙動だ。生の言語モデルは呼び出しの間にメモリを持たない。1つの会話の中で「覚えている」ことはすべてコンテキストウィンドウの中にあり、その会話が終われば消える。

**メモリ**とは、それを補うために外付けする仕組みのことだ。エージェントが何を引き継ぐべきか、それをどこに保存するか、そして適切な瞬間に適切な断片をどう取り戻すかを決めるシステム群を指す。2026年、これは脇道の課題ではなくエージェント設計の一級市民となり、独自のベンチマーク、フレームワーク、そして本格的な研究文献を持つに至った。このページはその地図だ。

## 押さえておくべき唯一の考え：コンテキスト ≠ メモリ

最もよくある混同は、大きなコンテキストウィンドウを「メモリ」だと考えることだ。違う。コンテキストウィンドウは**1ターン分の作業スペース**であり、呼び出しのたびにゼロから詰め直され、有限で、そして高価だ。加えて、その中でアテンションは劣化する（[コンテキストエンジニアリング](/docs/frontiers/context-engineering)で扱う「lost in the middle（中間で迷子になる）」効果）。

メモリは3つの点で異なる：

| | コンテキストウィンドウ | メモリ |
| --- | --- | --- |
| **寿命** | 1リクエスト | セッションをまたいで、数日、永遠に |
| **サイズ** | 固定のトークン上限 | 事実上無制限（外部ストア） |
| **コスト** | 毎ターン支払う | 書き込むとき一度だけ支払い、参照は安い |
| **アクセス** | すべてが常に視界にある | 選択的 — 関連するものだけを取り出す |

エージェントメモリのゲーム全体は、**適切な情報をこの2つの間で動かすこと**だ。永続的な事実はウィンドウの*外*に書き出して毎ターン支払わないようにし、この特定のステップに必要なときだけ*中*に引き戻す。この流れを正しくすれば、エージェントは常に数千の関連トークンしか保持しないコンテキストウィンドウで何週間も稼働できる。

## メモリの4つの種類

（緩やかに）認知科学から借りて、2026年のエージェントエコシステムは4つのカテゴリに収束した。4つすべてが必要になることは稀だが — 名前を付けておくと、すべてを中途半端にこなす一枚岩を作らずに済む。

  items={[

    {title: "エピソードメモリ", body: "起きた具体的な出来事を、タイムスタンプ付きで。『火曜にユーザーがチェックアウトが壊れていると言った、水曜にサポートが解決済みとマークした。』エピソードメモリは本質的に時間的だ — 順序と「いつ」が重要になる。エージェントがスナップショットではなく履歴について推論できるのはこれのおかげだ。"},
    {title: "意味メモリ", body: "永続的な事実と好み、それをいつ知ったかは剥ぎ取られている。『ユーザーはメートル法を好む。』『この顧客はエンタープライズプランだ。』意味メモリはおおむね非時間的で、エージェントが現在真であると信じていることを表す。それを知った出来事ではない。"},
    {title: "手続きメモリ", body: "何かのやり方 — 再利用可能なスキル、ワークフロー、学習したルーチン。実務上、4つの中で最も未成熟だ。Claude Code のようなツールでは、これは自動ストアではなく指示ファイル（CLAUDE.md）や再利用可能なスキルとして存在することが多い。"}
  ]}
/>

役に立つテスト：*「それはいつ起きた？」*と答えたくなるなら**エピソード**、*「何が真か？」*なら**意味**、*「やり方はこう」*なら**手続き**、そして次の数秒だけ重要なら**ワーキング**メモリで、まったく永続化する必要がない。

## 4つの保存パターン

*何を*覚えるべきかがわかったら、次は*どう*保存し取り出すかを選ぶ。おおむね複雑さの順で、4つの支配的なパターンがある。実際のシステムのほとんどは2つか3つを組み合わせる。

### 1. フルコンテキスト（全部を詰め込む）

履歴全体を保持し、毎ターン再送する。インフラゼロ、完璧な想起 — トークン上限、コストカーブ、あるいは「lost in the middle」にぶつかるまでは。短いアシスタントには向くが、長時間稼働するものには行き止まりだ。これは他のすべてのパターンが改善する土台となるベースラインだ。

### 2. ベクトル / RAG メモリ

各メモリを埋め込みとしてベクトルデータベースに書き込み、クエリ時に現在のターンを埋め込んで、最も類似する top-k のメモリを取り出す。これはドキュメントではなく会話履歴に向けた[検索拡張生成（RAG）](/docs/foundations/rag)だ。安価でスケーラブルで、事実や好みの意味的想起のデフォルトだ。

弱点：**時間的**あるいは**マルチホップ**な質問では、類似度 ≠ 関連度になる。「予算が削られた*後*、私たちは何を決めた？」は順序の質問であり、コサイン類似度には時間の感覚も、2つの事実をつなぐ感覚もない。

### 3. ナレッジグラフメモリ

メモリを**エンティティと関係**として保存する — ノードとエッジで、しばしばエッジにタイムスタンプを持つ。質問に答えるにはベクトルのあいまいマッチではなくグラフをたどる。これがマルチホップと時間的推論を扱える理由だ（「ユーザーが苦情を言ったアカウントの担当者を交代させたのは誰か？」）。Zep/Graphiti のようなフレームワークは、時間的ナレッジグラフを丸ごと売りにして構築された。代償は本物のエンジニアリングだ：抽出、エンティティ解決、そしてグラフを腐らせないための保守。

### 4. 圧縮と要約

稼働中の履歴を定期的に蒸留した要約に圧縮し、そこから続ける — 逐語的な想起を、より小さく安いウィンドウと引き換える。これは Claude Code の `/compact` がやっていることであり、多くのチャット製品の「自動要約」がやっていることだ。長期メモリの最も安価な形であり、実際に最初に必要になることが多い。リスク：要約は必要だった唯一の細部を静かに落としてしまう。数時間に及ぶ実行でこれがどう作用するかは[長時間稼働エージェントのハーネス](/docs/frontiers/long-running-agent-harnesses)を参照。

<Callout type="tip">
実際のシステムはこれらを重ねる。よくある2026年のスタック：稼働中の会話には<strong>圧縮</strong>、意味的事実には<strong>ベクトル</strong>、そして時間的/マルチホップのクエリが実際にトラフィックに現れたときにだけ、その上に<strong>グラフ</strong>。グラフが解決する痛みを感じるまでグラフを作るな。
</Callout>

## 大手3社はどうやっているか

いまや主要なアシスタントはどれも何らかのメモリを搭載している。それらは同じものではなく、違いが重要だ。

| 製品 | 何を覚えるか | どう動くか（おおまかに） |
| --- | --- | --- |
| **Claude** | 2層：あなたの好みのアプリレベルのメモリと、エージェント向けの開発者向け**メモリツール**。 | [Claude アプリのメモリ](/docs/claude-app/memory)はチャットをまたいで事実を保存する。API の[メモリツールとコンテキスト編集](/docs/api/memory-and-context-editing)は、エージェントがクライアント側のストアにメモを書き込み、古いツール結果を自動的に刈り込んで長時間の実行を生き延びさせる。 |
| **ChatGPT** | 「保存されたメモリ」（明示的な事実）に加え、過去のチャットへの参照。 | ユーザーが述べた事実と自動抽出された好みの混合で、後のターンでシステムコンテキストに注入される。ユーザーが編集でき、オン/オフを切り替えられる。 |
| **Gemini** | あなたのチャットから、また任意で、より広い Google アカウント面から引き出された個人的コンテキスト。 | 以前の会話の詳細を思い出し、プライバシー設定に従ってアカウントのコンテキストを使ってパーソナライズできる。 |

2つの要点。第一に、**コンシューマー向けメモリはおおむね意味的**だ — 好みと事実であって、完全なエピソードの再生ではない。第二に、エージェントを*構築している*なら、製品の組み込みメモリはあなたのメモリシステムではない。その層はあなたが所有し、Claude のメモリツールのようなプリミティブや外部フレームワークを使う。

<PromptCard title="生のモデルをメモを取るエージェントに変える（最も安い本物のメモリ）">
{`You have a file called MEMORY.md that persists between our sessions.

At the END of each session, append any durable facts worth keeping:
- my stable preferences (tools, formats, style)
- decisions we made and WHY
- open threads to resume next time

At the START of each session, read MEMORY.md first and use it.
Keep it under 30 lines — when it grows past that, consolidate and
delete anything stale. Never store secrets or credentials.`}
</PromptCard>

その1つのパターン — 永続的なメモを外部ファイルに書き、次回それを読み戻す — こそがエージェントメモリの80/20だ。以下のフレームワーク機構のほとんどは、まさにこれをより自動的に、よりスケーラブルにしたバージョンにすぎない。

## メモリを測る：LoCoMo ベンチマーク

測れないものは改善できないが、ベンチマークが登場するまでメモリは測りにくかった。最も引用されるのが **LoCoMo**（「LLM エージェントの超長期会話メモリの評価」）だ：非常に長いマルチセッションの会話 — 数十のセッションにわたる数百のターン — に、5種類の質問応答ペアが付く：シングルホップ、マルチホップ（セッションをまたぐ）、時間的推論、オープンドメイン、そして敵対的。

LoCoMo が明らかにするのは、設計の指針とすべきパターンだ：システムはシングルホップの事実想起では問題なくこなし、**時間的およびマルチホップ**の質問で崩れる。その失敗モードこそ、ナレッジグラフメモリが存在する理由だ — その2カテゴリを最も押し上げるパターンなのだ。自分のエージェントのメモリを評価するときは、マルチホップと時間的なケースを重く重み付けせよ。シングルホップの想起はほとんど何でもよく見せてしまう。

<VerifyNote lastVerified="2026-07-03" source="https://snap-research.github.io/locomo/">
LoCoMo は標準的な学術リファレンスだ（arXiv 2402.17753）。特定の LoCoMo スコアを主張するベンダーのリーダーボードは速く動き、しばしば自己申告だ — 見出しの数字を事実として扱う前に、論文の方法論と照合すること。
</VerifyNote>

## 結論

メモリはオンにする1つの機能ではない — それは**設計する流れ**だ：何がウィンドウを去るか、どこに保存されるか、そしてどう戻ってくるか。4種類のメモリに名前を付けて、すべてを1つの一枚岩で作らないようにせよ。あなたの問題を解決する最も安い保存パターンから始め、次のパターンの痛みを感じたときにだけ登れ。そして時間的とマルチホップのケースで測れ。シングルホップの想起は、すべてを実際より賢く見せてしまうからだ。

メモリは[コンテキストエンジニアリング](/docs/frontiers/context-engineering)のもう半分だ：コンテキストエンジニアリングは*このターン*にウィンドウを何が満たすかを決め、メモリはターンの*間*に何が生き残るかを決める。両者そろって、チャットボットと、あなたが長く付き合うほど良くなるエージェントとを分けるものになる。
