「常に最新」という約束は、魔法ではなくシンプルで正直な仕組みによって守られています。貢献する際にこれを支える方法を説明します。

## 普遍的なもの vs 変動するもの

- **普遍的なもの** — 変化しない概念（LLM とは何か、RAG とは何か）。日付は**不要**です。
- **変動するもの** — リリースごとに変わるもの。モデル名／ID、価格、制限、UI ラベル、ベータ／機能の提供状況、正確な CLI フラグなど。**日付と出典を必ず付与してください。**

## 変動しやすい事実に印を付ける

レベルバッジのすぐ下に `<VerifyNote>` コンポーネントを使います。

```mdx
<VerifyNote lastVerified="2026-06-20" source="https://docs.anthropic.com/...">
One line on what's volatile here and to confirm at the source.
</VerifyNote>
```

- **`lastVerified`** = *あなた*が実際に出典と照らし合わせて確認した日付。
- **`source`** = 権威ある公式ページ。

## モデルに関する事実をハードコードしない

モデル ID／価格／コンテキストは `data/models.json` の中**だけ**に存在し、[モデルページ](/docs/whats-new/models-and-pricing)で `<ModelTable />` によって描画されます。本文中ではモデル ID や価格を書く代わりに、**そこへリンク**してください。モデルの更新は、その JSON への 1 行の編集で済みます（最初の PR にうってつけです）。

## `needs-verification` への対応

事実が古くなっている可能性があるとき、自動化（および読者）は `needs-verification` ラベルの Issue を作成します。これを解決するには次のようにします。

1. 主張を公式の出典と照らし合わせて確認します。
2. 正しい場合 → `lastVerified` を今日の日付に更新します。
3. 変わっている場合 → 内容**と**日付を修正し、出典をリンクします。
4. モデルに関する事実の場合 → 本文ではなく `data/models.json` を更新します。

## 正直さの原則

:::tip 古い日付は機能のひとつ
数か月前の `lastVerified` は失敗を意味しません。むしろ*その*事実は再確認したほうがよいと読者に伝えます。自信たっぷりに間違った日付を示すより、正直に古い日付を示すほうが優れています。実際に再検証していない日付を更新してはいけません。
:::
