## Cos'è MCP (in una frase)

**MCP è uno standard aperto per connettere applicazioni AI a sistemi esterni.** Un client AI (come Claude) parla un solo protocollo; qualsiasi **server MCP** che lo parla può innestarsi e offrire al client nuove abilità. L'analogia ufficiale è una **porta USB-C per l'AI**: invece di un'integrazione su misura per ogni strumento, costruisci verso un solo connettore e funziona ovunque.

Anthropic ha introdotto e reso open source MCP, e fin dal primo giorno è arrivato con il **supporto per server MCP locali** così che Claude potesse connettersi a sistemi interni e dataset che girano sulla tua macchina.

<VerifyNote lastVerified="2026-06-28" source="https://modelcontextprotocol.io/">
I numeri di adozione, la data di revisione della spec, la governance (la gestione di MCP) e l'elenco esatto dei client che supportano MCP cambiano di frequente. Tratta ogni cifra specifica come deperibile e conferma su <a href="https://modelcontextprotocol.io/">modelcontextprotocol.io</a> e nell'annuncio di Anthropic. I fatti duraturi — MCP è uno standard aperto, client ↔ server, i server possono girare localmente, ampiamente adottato tra i client — sono stabili.
</VerifyNote>

Un server MCP può esporre tre tipi di cose al client:

- **Tool** — azioni che il modello può chiamare (leggere un file, eseguire una query, colpire un'API). Ha la stessa forma del [tool use](/docs/api/tool-use) nativo del modello: una capacità con nome e input tipizzati.
- **Resource** — dati che il client può leggere (file, record, documenti).
- **Prompt** — template di prompt riutilizzabili e parametrizzati che il server offre.

Per questa pagina il titolo sono i **tool**: un server MCP locale è il modo in cui consegni a Claude una nuova capacità privata.

## Perché questa è la sinergia che vuoi

Ciò che la gente intende con "Claude come cervello, il mio computer come corpo" è esattamente la divisione client/server di MCP:

- **Claude è l'orchestratore.** Legge il tuo obiettivo, decide quale tool chiamare, lo chiama, legge il risultato e decide lo step successivo.
- **I server MCP locali sono le capacità.** Ciascuno è un piccolo programma che *tu* esegui sulla *tua* macchina: un server filesystem, un server database, un wrapper attorno a uno script locale, persino un server che chiama un [modello locale](/docs/models/local-ai-agents) o affida il lavoro a un agent locale.

Poiché il server gira localmente e Claude gli parla tramite lo standard input/output della tua macchina, **i tuoi dati e le tue azioni restano sul tuo hardware** — il modello orchestra, ma le letture di file, l'SQL, l'esecuzione degli script avvengono localmente. È la versione a tutela della privacy del lavoro agentico: cervello nel cloud (o anche locale), mani rigorosamente sulla tua macchina.

E poiché MCP è **un solo standard aperto adottato tra molti client**, un server che costruisci non è vincolato a una singola app. Lo stesso server locale che colleghi a [Claude Code](/docs/claude-code/what-is-claude-code) può essere usato da altri client che parlano MCP — costruisci una volta, riusa ovunque.

## L'architettura locale

Il quadro utile più piccolo ha due parti e un canale:

- **Client MCP (host).** L'app AI — per esempio [Claude Code](/docs/claude-code/what-is-claude-code) o Claude Desktop. Contiene il modello e decide quali tool chiamare.
- **Server MCP.** Un processo separato che pubblicizza tool/resource/prompt e li esegue quando richiesto.
- **Transport.** Per i server locali il client avvia il server come sottoprocesso e comunicano su **stdio** (standard input/output) usando messaggi JSON-RPC. Niente rete, niente porta — solo una pipe locale tra due processi sulla tua macchina. (Esistono anche server remoti, su HTTP; questa pagina riguarda il caso locale, stdio.)

```
┌─────────────────────────┐         stdio (JSON-RPC)        ┌──────────────────────────┐
│  MCP CLIENT / HOST       │ ──── launches as subprocess ──▶ │  LOCAL MCP SERVER         │
│  Claude Code / Desktop   │ ◀── tools/resources/prompts ─── │  filesystem · db · script │
│  (the model = the brain) │                                 │  · local model / agent    │
└─────────────────────────┘                                  └──────────────────────────┘
                                                              runs on YOUR machine, YOUR data
```

Il modello non tocca mai direttamente il tuo disco — chiede al server locale, e il server (eseguito con i tuoi permessi) fa il lavoro e restituisce un risultato.

## Aggiungere un server MCP locale a Claude

La via più rapida è il comando `claude mcp add` di Claude Code, che registra un server che il client avvierà su stdio. La forma generale è un nome, poi il comando che avvia il server.

  {title: "Conferma che si è caricato", body: "Esegui `claude mcp list` per vedere i server registrati, poi avvia una sessione e chiedi a Claude quali tool ha ora. I tool del server dovrebbero comparire come capacità chiamabili."},
  {title: "Usalo in un compito", body: "Dai a Claude un obiettivo che richiede la nuova capacità (\"leggi i file sotto ./notes e riassumili\"). Claude chiama i tool del server; il lavoro avviene localmente."},
]} />

Un `claude mcp add` minimale per un server filesystem locale (tutto ciò che segue `--` è il comando di avvio):

<PromptCard title="Aggiungere un server MCP filesystem locale a Claude Code">{`claude mcp add filesystem -- npx -y @modelcontextprotocol/server-filesystem /path/to/allowed/dir`}</PromptCard>

Se preferisci modificare direttamente la config, un server stdio è dichiarato dal comando che lo avvia. La forma è questa:

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/allowed/dir"]
    }
  }
}
```

<VerifyNote lastVerified="2026-06-28" source="https://modelcontextprotocol.io/">
Gli esatti sottocomandi `claude mcp`, gli scope di configurazione, la posizione del file di config e i nomi precisi dei pacchetti dei server di riferimento evolvono. Conferma la sintassi attuale nella <a href="/docs/claude-code/mcp">guida MCP di Claude Code</a> e nel repo dei server di riferimento prima di affidarti a un flag o nome di pacchetto specifico.
</VerifyNote>

## Costruire un piccolo server MCP

Un server è solo un programma che usa un SDK MCP, dichiara un tool e avvia un transport stdio. Ecco l'idea intera in TypeScript — un tool che somma due numeri (sostituisci il corpo con "esegui il mio script", "interroga il mio DB", "chiama il mio modello locale"):

```bash
npm install @modelcontextprotocol/sdk zod
```

```ts
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({ name: "my-local-tools", version: "0.1.0" });

// Declare ONE tool. Inputs are typed; the body runs locally with your privileges.
server.registerTool(
  "add",
  {
    description: "Add two numbers",
    inputSchema: { a: z.number(), b: z.number() },
  },
  async ({ a, b }) => ({
    content: [{ type: "text", text: String(a + b) }],
  }),
);

// Talk to the client over stdio — this is what makes it a LOCAL server.
await server.connect(new StdioServerTransport());
```

Registralo con Claude esattamente come qualsiasi altro server stdio — punta il comando di avvio al tuo file:

<PromptCard title="Aggiungere il tuo server locale a Claude Code">{`claude mcp add my-local-tools -- node /abs/path/to/server.js`}</PromptCard>

Ecco il loop completo: Claude (cervello) vede un nuovo tool, lo chiama quando serve, e il tuo codice (mani) gira localmente e restituisce il risultato. Poiché parla lo stesso protocollo aperto, questo server funziona anche in altri client MCP senza modifiche.

<Callout type="warning" items={["I server MCP girano con i tuoi privilegi e possono compiere azioni reali — installa solo server di cui ti fidi, e sta' attento alla prompt-injection via risultati dei tool."]} />
