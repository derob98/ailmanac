## La mappa concettuale in 60 secondi

Se leggi una sola sezione, leggi questa. Ecco come le cose che conosci in Claude si allineano con il mondo di Kimi:

| In Claude lo chiami… | Nel mondo di Kimi è… | Stessa idea? |
| --- | --- | --- |
| Claude.ai (l'app di chat) | L'**app Kimi** (kimi.com / mobile) | Sì — la superficie consumer/assistente |
| Selettore di modello (Opus / Sonnet / Haiku) | Varianti K2 — **K2-Instruct** (agentico veloce) e **K2 Thinking** (ragionamento profondo) | Sì — scegli tra velocità e profondità di ragionamento |
| Extended thinking | Chain-of-thought interlacciato + tool call di **K2 Thinking** | Sì — qui il ragionamento è una variante, non solo un interruttore |
| Anthropic Messages API | L'API nativa di Moonshot **e** un endpoint compatibile con Anthropic | In parte — il layer di compatibilità parla il *tuo* dialetto |
| Tool use | **Native tool calling** di Kimi (agent-first per design) | Sì — stesso loop dichiara→chiama→esegui→restituisci |
| Pesi chiusi, solo hosted | **Pesi aperti** su Hugging Face (Modified MIT) | No — questo è il punto chiave; puoi fare self-hosting |
| Claude Code | Claude Code **puntato su Kimi**, o la Kimi CLI di Moonshot | Perlopiù — stesso harness, modello diverso dietro |

La riga di gran lunga più importante è la penultima: **Kimi K2 è open-weight.** Tutto ciò che ne deriva — self-hosting, prezzo basso, licenza permissiva — discende da lì.

## Cos'è davvero Kimi K2

Kimi K2 è un modello **Mixture-of-Experts**: circa **1T di parametri totali con ~32B attivi per token** (384 esperti, 8 selezionati per token). È stato pre-addestrato su ~15,5T token ed è rilasciato sotto una **Modified MIT License** — pesi genuinamente aperti che puoi scaricare, ispezionare ed eseguire. Moonshot lo ha costruito agent-first: la model card lo descrive come "specificamente progettato per l'uso di strumenti, il ragionamento e la risoluzione autonoma dei problemi."

Due varianti contano di più per te:

- **K2-Instruct** — il modello agentico/chat drop-in. Contesto 128K. Auto-riportato **65,8% su SWE-bench Verified**, 53,7% su LiveCodeBench, 89,5% su MMLU. Questa è la tua daily driver in stile Sonnet.
- **K2 Thinking** — l'agente di ragionamento. Interlaccia chain-of-thought con function call end-to-end e rimane stabile attraverso **200–300 tool call sequenziali** — il tratto per cui la gente va in visibilio per gli agenti a lungo orizzonte. INT4 nativo, **contesto 256K**. Auto-riportato **71,3% su SWE-bench Verified**, 44,9% su Humanity's Last Exam (con strumenti), 60,2% su BrowseComp. Questo è il tuo analogo di Opus con extended thinking.

## La feature killer per gli utenti Claude: punta Claude Code su Kimi

A differenza della maggior parte delle alternative, non devi lasciare il tuo harness. Moonshot espone un **endpoint compatibile con Anthropic** (`/anthropic/v1/messages`), quindi Claude Code — che parla l'Anthropic Messages API — può dialogare con Kimi K2 sovrascrivendo due variabili d'ambiente.

## Oltre Claude Code: l'API raw e il self-hosting

Se stai scrivendo la tua integrazione invece di guidare Claude Code, hai tre strade:

- **Endpoint compatibile con Anthropic** — riutilizza il tuo client in stile Claude/Messages cambiando la base URL e la chiave (come sopra). Frizione minima se il tuo codice è già in stile Anthropic.
- **API nativa di Moonshot** — Moonshot offre anche una superficie compatibile con OpenAI, quindi anche il codice in stile OpenAI migra con uno swap di base-URL + chiave. Comodo se il tuo stack proviene dal lato OpenAI.
- **Self-host dei pesi** — scarica da Hugging Face e servi con un motore di inferenza che supporti il parsing nativo degli strumenti di Kimi (vLLM, SGLang e affini). Questo è l'endgame di privacy/costi: nessun token lascia la tua infrastruttura. Vedi [Eseguire modelli localmente con Ollama](/docs/models/run-models-locally-ollama) e [Stack AI privato e locale](/docs/models/private-local-ai-stack) per il percorso locale.

## Cosa si trasferisce invariato

Quasi tutta la tua maestria con Claude si trasferisce, perché Kimi K2 è agent-first e tool-native come Claude:

- **Abitudini di prompting.** Istruzioni chiare, vincoli espliciti, esempi e role framing funzionano tutti. Vedi [Basi del prompting](/docs/prompting/basics) e [Portare i prompt tra i modelli](/docs/models/porting-prompts-across-models).
- **Disciplina nell'uso degli strumenti.** Il loop dichiara→chiama→esegui→restituisci ha la stessa forma del [Tool Use](/docs/api/tool-use) di Claude. Kimi è costruito per questo e regge su catene di strumenti molto lunghe.
- **Context engineering.** Contesto stretto, buon retrieval e input strutturati contano allo stesso modo — probabilmente di più nelle run agentiche lunghe. Vedi [Context Engineering](/docs/frontiers/context-engineering).
- **Capacità di guidare gli agenti.** Definire l'ambito dei task, revisionare i diff e mantenere le run sui binari si trasferiscono direttamente a qualsiasi harness — incluso Claude Code puntato su Kimi. Vedi [Cos'è Claude Code?](/docs/claude-code/what-is-claude-code) e [Harness per agenti a lunga esecuzione](/docs/frontiers/long-running-agent-harnesses).

## Quando Kimi K2 tende a brillare

- **Run agentiche a lungo orizzonte.** Il tratto di punta è la stabilità attraverso centinaia di tool call sequenziali — punta su K2 Thinking quando un task è una maratona di uso di strumenti, non una singola risposta.
- **Coding attento ai costi su larga scala.** Solidi risultati auto-riportati su SWE-bench con economics da open-weight lo rendono un bersaglio A/B naturale per agenti di coding ad alto volume.
- **Privacy e self-hosting.** I pesi aperti significano che puoi eseguirlo interamente sul tuo hardware — l'unica cosa che nessun modello di frontiera chiuso può offrire.
- **Mantenere la memoria muscolare di Claude Code.** Quando vuoi un modello più economico o self-hosted *senza* riapprendere un flusso di lavoro, lo swap di due variabili è difficile da battere.

La risposta onesta il più delle volte: **quello che vince la tua eval sul task che conta — e quello che i tuoi vincoli di privacy e budget consentono.** La skill è portabile; la configurazione e l'eval sono il vero lavoro. Per il modo neutro rispetto al provider di scegliere, vedi [Scegliere un modello](/docs/models/choosing-a-model) e [Quanto costa l'AI tra i provider](/docs/models/what-ai-costs-across-providers).
