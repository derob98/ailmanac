Per un anno, la "memoria di Claude" era un singolo paragrafo in evoluzione che il modello riscriveva ogni giorno — buono per una bio, pessimo per qualsiasi cosa con struttura. Tra **giugno e luglio 2026** Anthropic l'ha sostituito con un modello basato su voci: decine di piccole note categorizzate che Claude legge e scrive in tempo reale. La stessa idea alimenta l'app *e* l'API di memoria dei Managed Agents. Questa pagina le collega perché tu capisca entrambe le superfici da un unico modello mentale.

<VerifyNote lastVerified="2026-07-24" source="https://platform.claude.com/docs/en/release-notes/overview">
L'header beta `agent-memory-2026-07-22`, gli eventi webhook memory_store e il supporto ai modelli Dreams sono cambiati tutti nelle ultime quattro settimane. Verifica lo stato beta attuale e i prezzi sulle release notes della piattaforma prima di andare in produzione.
</VerifyNote>

## Il modello mentale che unifica entrambi i lati

```mermaid
flowchart LR
  subgraph App["Claude.ai (Free / Pro / Max / Team / Enterprise)"]
    U[L'utente chatta] --> W1[Claude scrive una voce]
    W1 --> S["Impostazioni › Memoria<br/>(visualizza, modifica, elimina per voce)"]
  end
  subgraph API["Managed Agents (memory_stores)"]
    A[L'agente gira] --> W2[L'agente crea una memoria<br/>a un path]
    W2 --> L["GET /v1/memory_stores/*/memories<br/>lista per path_prefix"]
  end
  App -. stessa idea di "voci piccole, categorizzate" .- API
```

Entrambi i lati ora funzionano allo stesso modo: **molte voci piccole in posizioni denominate**, ognuna riscrivibile in isolamento. Il vecchio modello (un unico blob per utente, riscritto in blocco) è sparito nell'app e non è mai stato la superficie dell'API. Una volta interiorizzato *"voci ai path, elencate per prefisso, modificate una alla volta"*, entrambe le superfici smettono di sembrare feature separate.

### Perché il cambio di forma è importante

Il vecchio riassunto giornaliero aveva un'unica modalità di fallimento: contesti non correlati sbordavano l'uno sull'altro. Correggevi un fatto sul tuo progetto di lavoro e Claude poteva sovrascrivere qualcosa sul tuo stile di scrittura. Voci piccole e categorizzate risolvono questo problema strutturalmente — ogni modifica è confinata a una corsia.

Tre conseguenze che la maggior parte degli utenti si perde:

- **Il recupero ora è mirato, non associativo.** Claude può tirare fuori "le tue preferenze di comunicazione" senza trascinarsi dietro il resto del tuo contesto. I prompt che prima dovevano dire "ricorda che preferisco X, non Y" per sfondare il riassunto ora colpiscono direttamente una voce esistente.
- **Puoi potare chirurgicamente.** Elimina una singola voce sbagliata nelle Impostazioni invece di cancellare tutto e ri-insegnare a Claude chi sei. La tassa del "brucio tutto" per correggere un errore è sparita.
- **Le categorie agiscono come uno schema invisibile.** Quando Claude decide *dove* appartiene una voce (es. "stile di comunicazione" vs. "progetto attuale"), sta scegliendo implicitamente se quel fatto verrà richiamato in future chat personali, future chat di lavoro, o entrambe. Nominare bene le cose fin da subito guida un recupero migliore in seguito.

## Parte 2 — Nell'API: memory_stores e il nuovo header beta

La piattaforma Managed Agents espone la memoria come **memory store** — collezioni di memorie indirizzate per path, elencate con un `path_prefix` e paginate con cursori. Il **2 luglio 2026** Anthropic ha rilasciato `agent-memory-2026-07-22`, un header beta che cambia la semantica abbastanza da rompere le migrazioni naïve. Il **22 luglio 2026** il precedente header `managed-agents-2026-04-01` ha adottato lo stesso comportamento di lista — quindi le regole "nuove" ora sono le *uniche* regole sugli endpoint dei memory store.

### I quattro cambi di comportamento che devi conoscere

| # | Cosa è cambiato | Vecchio comportamento | Nuovo comportamento sotto `agent-memory-2026-07-22` |
|---|---|---|---|
| 1 | Ordine della lista | Controllato dal client via `order_by` / `order` | **Ordine stabile, definito dal server.** `order_by` e `order` sono ignorati — ordina client-side se serve |
| 2 | Parametro `depth` | Intero libero | Accetta solo `0`, `1`, o omesso. Qualsiasi altro valore restituisce **400** |
| 3 | Match di `path_prefix` | Match di substring | Deve finire con `/`, matcha **solo segmenti interi di path** |
| 4 | Cursori di paginazione | Interscambiabili | I cursori emessi senza il nuovo header **non sono validi con esso**. Riparti da pagina 1 quando lo adotti |

Altre due insidie:

- **Collisione di header.** Sugli endpoint dei memory store, `agent-memory-2026-07-22` sostituisce `managed-agents-2026-04-01`. Inviare entrambi restituisce **400**. Migra solo le chiamate ai memory store; lascia gli altri endpoint di managed-agents sull'header originale.
- **Gli SDK sono già passati.** Python 0.116.0, TypeScript 0.110.0, Go 1.56.0, Java 2.48.0, Ruby 1.55.0, PHP 0.36.0, C# 12.35.0, e la CLI 1.16.0 inviano tutti `agent-memory-2026-07-22` automaticamente sulle chiamate ai memory store. Se il tuo codice passa `betas` esplicitamente, **sostituisci** lì `managed-agents-2026-04-01` — non aggiungerli entrambi.

### Webhook: reagisci al ciclo di vita del memory store senza polling

Rilasciati il **22 luglio 2026** insieme al default dell'header: **tre tipi di evento webhook `memory_store.*`** ora coprono il ciclo di vita del memory store, e quattro eventi `environment.*` coprono l'ambiente sandbox. Questo è il pezzo che la maggior parte degli integratori si perde — ora puoi reagire alla creazione, all'aggiornamento o alla rimozione di un memory store da parte di una run dell'agente invece di fare polling su `GET /v1/memory_stores` a intervalli. Iscriviti all'endpoint webhook di Managed Agents e instrada gli eventi nel tuo audit trail o dashboard.

### Dreams: riorganizza la memoria in background

Annunciato nelle release notes del 6 maggio 2026 ed esteso il 10 luglio per supportare Claude Fable 5 e Sonnet 5, **Dreams** è una research preview che legge un memory store esistente *più* i transcript delle sessioni passate e produce un **memory store di output riorganizzato**: duplicati fusi, voci obsolete sostituite, nuovi insight fatti emergere. Gli endpoint sono protetti dall'header beta `dreaming-2026-04-21`; l'accesso è su richiesta. Trattalo come l'analogo lato API del recap mensile dell'app — una passata di riflessione schedulata che impedisce allo store di sedimentare.

## App vs. API — una tabella

| Aspetto | App Claude.ai | API Managed Agents |
|---|---|---|
| Unità di storage | Voce categorizzata | Memoria a un path |
| Raggruppamento | Categoria (contesto professionale, preferenze di comunicazione, progetti in corso, …) | Segmenti di path (`/projects/acme/…`) |
| Lettura/scrittura | In tempo reale durante la chat | Chiamate a tool dell'agente durante una sessione |
| Editabile dall'utente | Sì, per voce in Impostazioni › Memoria | Sì, via endpoint dei memory store |
| Cancellazione | Per voce, o tutto | Per memoria, per prefisso, o eliminando lo store |
| Riflessione | Recap mensile (Impostazioni › Reflect) | Dreams (research preview) |
| Isolamento | Chat in incognito | Selezione del memory store per-agente, per-sessione |
| Disponibilità | Free, Pro, Max, Team, Enterprise (verifica per piano) | Managed Agents (beta pubblica) |

Dove questo si colloca nel resto di AILmanac:

- [Memoria e Context Editing](/docs/api/memory-and-context-editing) — il memory tool *lato client*, una bestia diversa: lo storage lo implementi tu.
- [Memoria tra le chat](/docs/claude-app/memory) — la panoramica per principianti sulla memoria dell'app.
- [Architetture di memoria degli agenti](/docs/frontiers/agent-memory-architectures) — la teoria della memoria episodica vs. semantica che fa capire perché le voci categorizzate battono un riassunto in rolling.
- [Managed Agents](/docs/api/managed-agents) — l'harness che ospita la superficie dei memory store descritta qui.
