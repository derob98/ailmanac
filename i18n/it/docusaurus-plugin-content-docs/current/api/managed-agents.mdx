Se [costruire il tuo loop dell'agent](/docs/api/building-agents) è più infrastruttura di quanta tu voglia gestire, un'opzione di agent **gestito** (ospitato da Anthropic) esegue il loop al posto tuo — così ti concentri sul *lavoro* dell'agent, non sull'impalcatura delle sessioni, sui retry e sullo stato.

## Cosa ti offre "gestito"

Invece di costruire e ospitare il loop a mano, ottieni mattoni ospitati come:

- **Sessioni** — esecuzioni persistenti dell'agent che puoi riprendere.
- **Ambienti** — un luogo in cui l'agent può operare con i suoi strumenti.
- **Memoria** — stato che si trasporta tra turni/esecuzioni senza che tu debba collegare un database.
- **Scheduling** — agent che girano secondo una pianificazione, senza supervisione.

(I nomi esatti e la superficie evolvono — considera la [documentazione ufficiale](https://docs.anthropic.com/en/docs/agents-and-tools) come autorevole.)

## Quando scegliere gestito vs personalizzato

| Scegli **gestito** quando… | Scegli un **loop personalizzato / SDK** quando… |
|---|---|
| Vuoi che hosting, stato e scheduling siano gestiti per te | Hai bisogno del pieno controllo su loop e strumenti |
| Stai prototipando velocemente | Hai requisiti rigorosi di infrastruttura/compliance personalizzati |
| La semplicità operativa conta più del controllo | Stai integrando in profondità nel tuo stack |

È uno spettro — chiamata singola → workflow → agent personalizzato (SDK) → gestito. Parti dalla soluzione più semplice che il task consente; sali di livello solo quando ne hai bisogno.

## Valgono gli stessi guardrail

Ospitato o meno, un agent autonomo compie comunque azioni. Mantieni **privilegio minimo**, **costo/iterazioni limitati** e **approvazione umana per i passi rischiosi** — vedi [Mettere in sicurezza gli agent](/docs/security/securing-agents) e [Irrobustire le esecuzioni autonome](/docs/security/hardening-autonomous-runs).
