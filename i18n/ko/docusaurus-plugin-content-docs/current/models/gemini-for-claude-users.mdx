## 60초 개념 대응표

한 섹션만 읽는다면 이걸 읽으세요. 당신이 Claude에서 아는 것들이 Gemini와 어떻게 대응하는지는 다음과 같습니다:

| Claude에서 부르는 이름… | Gemini 세계에서는… | 같은 개념? |
| --- | --- | --- |
| Claude.ai (채팅 앱) | gemini.google.com의 **Gemini 앱** | 예 — 소비자/어시스턴트 표면 |
| Claude 프로젝트 (워크스페이스 + 파일 + 지침) + 재사용 가능한 커스텀 스타일 | **Gems** (지침 + 지식 파일을 갖춘, 저장되고 재사용 가능한 어시스턴트) | 유사 — 아래 구분 참고 |
| 계정 환경설정 / "이렇게 답해줘" | Gem 지침 + 계정 설정 | 예 — 같은 개념 |
| 아티팩트 / 파일 컨텍스트 | Gem에 첨부된 **지식 파일** (Google Drive에서도 가능) | 유사 |
| 모델 선택기 (Opus / Sonnet / Haiku) | Gemini의 모델 선택기 (깊이를 위한 **Pro**, 속도를 위한 **Flash**) | 예 — 같은 개념, 다른 이름 |
| 콘솔 / API + 프롬프트 워크벤치 | **Google AI Studio** + **Gemini API** | 예 — 프로토타이핑하고 키를 얻는 개발자 표면 |
| 도구 사용 / 함수 호출 (API) | **함수 호출(Function calling)** | 예 — 같은 루프, 다른 JSON 세부사항 |

핵심은: 배워야 할 *새로운 개념*이 거의 없다는 것입니다. 대부분 당신이 이미 이해하는 것들을 **이름 바꾸고 위치 옮기는** 일입니다 — 여기에 Gemini가 Claude보다 정말로 더 나아간 두 영역이 더해집니다: **구글 생태계 통합**과 **네이티브 멀티모달 + 매우 큰 컨텍스트**.

## Gems는 당신이 이미 바라던 "재사용 가능한 Claude"입니다

**Gem**은 한 번 만들어 재사용하는 커스터마이즈 가능한 어시스턴트입니다: 이름을 붙이고, 지침을 쓰고, 선택적으로 참조할 지식 파일을 첨부합니다. Claude 시스템 프롬프트를 저장해 본 적이 있거나 참조 문서를 중심으로 Claude 프로젝트를 꾸려본 적이 있다면, Gem은 그 본능을 하나의 객체로 담은 것입니다. <VerifyNote lastVerified="2026-06-28" source="https://support.google.com/gemini/answer/15235603">구글은 Gem을 지침(흔히 페르소나, 작업, 컨텍스트, 형식으로 구성됨)에 선택적으로 업로드한 지식 파일(PDF, Word/Google 문서, Drive 파일)을 더해 정의되는 커스터마이즈 가능한 어시스턴트로 설명합니다. 정확한 생성 흐름, 파일 한도, 지원 형식은 바뀌므로 — 구글의 최신 도움말 페이지에서 확인하세요.</VerifyNote>

구글이 권장하는 지침 구조는 Claude 사용자에게 익숙하게 느껴질 것입니다. 그저 좋은 프롬프팅이기 때문입니다:

<Steps items={[
  {title: "페르소나", body: "Gem이 누구인지 — 그 역할과 목소리. Claude 시스템 프롬프트에서 쓰는 것과 같은 'You are a…' 구성."},
  {title: "작업", body: "무엇을 해야 하는지. 목표를 명확한 한 문장으로 진술하세요."},
  {title: "컨텍스트", body: "가정해야 할 배경 — 그리고 참조할 수 있는 지식 파일, Google Drive의 실시간 문서 포함."},
  {title: "형식", body: "출력의 형태: 표, 목록, JSON, 구조화된 문서. Claude에게 특정 형식을 요청하는 것과 똑같습니다."},
]} />

제대로 짚어둘 미묘한 점: Gem은 **지침 + 참조 자료를 하나의 재사용 가능한 어시스턴트로** 패키징합니다. Claude 프로젝트의 지침과 파일과 비슷하지만, 단지 당신의 사적인 워크스페이스가 아니라 팀이 실행할 수 있는 *공유 가능한 전문가*로 구성됩니다. Gem이 Drive 파일을 참조할 수 있을 때는 **가장 최신 버전**을 읽는 경향이 있습니다 — 채팅에 정적인 스냅샷을 붙여넣는 것과는 작지만 실질적인 차이입니다. <VerifyNote lastVerified="2026-06-28" source="https://workspace.google.com/blog/product-announcements/5-ways-to-boost-your-teams-productivity-with-the-gemini-app-featuring-new-sharable-gems">구글은 Workspace 구독자를 위한 공유 가능한 Gems와 Drive 파일을 (최신 버전으로) 참조하는 기능을 설명해 왔습니다. 가용성, 공유 범위, 파일 한도는 바뀌므로 — 구글의 최신 발표를 확인하세요.</VerifyNote>

## Workspace 통합이 진짜 헤드라인입니다

여기가 Gemini가 당신의 Claude 습관과 가장 크게 갈라지는 지점이며, 알맞은 작업에 대해서는 보통 *좋은* 방향입니다. Gemini 앱은 Google Workspace — Gmail, 문서, Drive, 캘린더 등 — 에 연결되어 있어서, 당신이 복사-붙여넣기하지 않아도 당신의 구글 데이터를 가로질러 컨텍스트를 가져오고 작업을 수행할 수 있습니다. <VerifyNote lastVerified="2026-06-28" source="https://knowledge.workspace.google.com/admin/gemini/google-workspace-with-gemini">구글은 Gemini가 Workspace 앱(Gmail, 문서, Drive 등)과 통합되어 콘텐츠를 노출하고 요약한다고 설명합니다. 어떤 앱이 연결되는지, 그리고 그 접근에 대한 관리자/계정 제어는 바뀌므로 — 구글의 최신 Workspace 문서에서 확인하세요.</VerifyNote>

Claude 사용자에게 사고의 전환은 이것입니다: *자료를 모델로 가져오는* 대신, 종종 *이미 당신의 구글 계정에 있는 자료를 모델이 가리키게* 할 수 있습니다. 이는 프롬프팅을 바꾸기보다 워크플로를 바꿉니다. 프롬프트는 여전히 "이걸 요약해, 저걸 추출해, 이걸 초안 잡아"이지만 — 그 "이것"이 붙여넣기가 아니라 실시간 Drive 문서나 받은편지함의 스레드일 수 있습니다.

]} />

## 네이티브 멀티모달 + 거대한 컨텍스트 — 이걸 적극 활용하세요

두 가지 능력은 Gemini가 습관적인 Claude 워크플로보다 종종 정말로 선택할 만한 지점입니다:

- **네이티브 멀티모달 입력.** Gemini 모델은 텍스트, 이미지, 오디오, 영상을 하나의 요청 안에서 함께 받도록 만들어졌습니다 — 이미지가 그냥 붙여진 게 아닙니다. 당신의 작업이 영상을 *보거나*, 오디오를 *듣거나*, 여러 미디어를 한꺼번에 놓고 추론하는 것이라면, 그건 Gemini의 강점 지대입니다. <VerifyNote lastVerified="2026-06-28" source="https://ai.google.dev/gemini-api/docs/vision">구글은 Gemini가 여러 입력 양식(텍스트, 이미지, 그리고 — 많은 모델의 경우 — 오디오와 영상)을 하나의 요청에서 받는다고 문서화합니다. 특정 모델이 어떤 양식을 지원하는지는 모델마다 바뀌므로 — 구글의 최신 모델 및 비전 문서에서 확인하세요.</VerifyNote>

- **매우 큰 컨텍스트 창.** 많은 Gemini 모델은 극도로 큰 입력을 받습니다 — 구글은 이를 코드베이스 전체, 여러 개의 긴 문서, 또는 몇 시간짜리 미디어를 한 번에 처리하여 청킹이나 검색 우회책의 필요를 줄이는 것으로 설명합니다. *정확한* 토큰 상한은 변동성이 크고 모델마다 다르므로 하드코딩하지 마세요. 오래가는 요점은 "전부 다 집어넣기"가 당신이 익숙한 것보다 여기서는 더 실행 가능한 전략이라는 점입니다. <VerifyNote lastVerified="2026-06-28" source="https://ai.google.dev/gemini-api/docs/long-context">구글은 많은 Gemini 모델이 큰 컨텍스트 창을 가진다고 설명하며(100만 토큰 이상을 처리할 수 있는 모델을 인용함), 긴 문서, 큰 코드베이스, 긴 영상/오디오를 하나의 요청에서 가능하게 합니다. 정확한 모델별 한도는 바뀌므로 — 구글의 최신 롱컨텍스트 문서에서 확인하세요.</VerifyNote>

조정할 습관: Claude에서는 시도조차 하기 전에 본능적으로 잘라내고, 청킹하고, 검색을 구축할 수 있습니다. 큰 컨텍스트를 가진 Gemini 모델에서는 종종 **문서(또는 영상) 전체를 하나의 프롬프트에 먼저 넣어보고**, 실제 한도에 부딪혔을 때만 최적화하는 편이 낫습니다.

## AI Studio + Gemini API — 개발자 표면

채팅에서 구축으로 넘어갈 때, Claude 사용자에게는 깔끔한 유비가 있습니다:

- **Google AI Studio**는 프로토타이핑 워크벤치입니다 — 프롬프트를 테스트하고, 멀티모달 입력을 시도하고, 파라미터를 조정하고, **API 키를 얻는** 가장 빠른 방법. 무언가를 연결하기 전에 실험하러 가는 곳이라고 생각하세요. <VerifyNote lastVerified="2026-06-28" source="https://ai.google.dev/aistudio">구글은 AI Studio를 Gemini로 프로토타이핑하고, 멀티모달 입력을 테스트하고, 파라미터를 조정하고, API 키를 얻는 개발자 환경으로 자리매김합니다. 정확한 기능과 무료 등급 조건은 바뀌므로 — 구글의 최신 AI Studio 페이지에서 확인하세요.</VerifyNote>
- **Gemini API**는 프로토타이핑을 마친 뒤 당신의 코드에서 호출하는 것입니다 — 콘솔 플레이그라운드에서 프로덕션 호출로 옮겨가는 것에 해당합니다.

똑바로 정리해둘 하나의 구성 차이: **Gemini 앱**(gemini.google.com)은 Workspace 통합을 갖춘 다듬어진 *어시스턴트*이고, **AI Studio**는 모델 동작을 테스트하고 키를 얻는 *개발자* 표면입니다. 서로 다른 문이며, Claude 사용자는 가끔 Claude.ai를 API 콘솔과 혼동하듯 이 둘을 혼동합니다. 일에 맞춰 문을 고르세요: 어시스턴트 작업 → 앱; 구축 → AI Studio 다음 API.

### 함수 호출 — 같은 형태, 다른 JSON 세부사항

Claude에서 [도구 사용](/docs/api/tool-use)을 해봤다면, Gemini의 **함수 호출**은 즉시 익숙하게 느껴질 것입니다. *루프가 같기* 때문입니다:

]} />

동일한 것: 기술 → 모델 요청 → 당신이 실행 → 결과 반환 → 답변의 루프, 그리고 *모델은 결코 당신의 코드를 실행하지 않는다*는 원칙. Gemini는 또한 **병렬** 호출(한 턴에 여러 독립 함수)과 **합성적** 호출(나중 호출이 이전 결과에 의존하는 함수 체이닝)을 지원합니다 — Claude에서 이미 아는 개념들입니다. <VerifyNote lastVerified="2026-06-28" source="https://ai.google.dev/gemini-api/docs/function-calling">구글은 이름/설명/JSON 스키마 파라미터로 함수를 선언하는 것과 정의 → 모델 호출 → 로컬 실행 → 결과 반환 루프, 그리고 병렬 및 합성적(순차) 호출을 문서화합니다. 정확한 필드 이름, 응답 형태, 옵션은 바뀌므로 — 구글의 최신 함수 호출 가이드에서 확인하세요.</VerifyNote>

**다르고** 복사-붙여넣기하면 걸려 넘어질 것: 정확한 JSON 필드 이름, 함수 호출과 결과가 메시지 목록에서 어떻게 표현되는지, 파라미터/스키마 관례, 그리고 구체적인 엔드포인트 형태. 제공업체 전반의 규칙으로서, **프롬프팅과 아키텍처는 이어지지만 와이어 포맷 세부사항은 이어지지 않습니다** — 언제나 제공업체별 스키마를 다시 확인하세요. ([Claude 사용자를 위한 ChatGPT](/docs/models/chatgpt-for-claude-users)에서 넘어오는 경우에도 같은 주의가 적용됩니다: 루프는 보편적이고, JSON은 그렇지 않습니다.)

## 무엇이 그대로 이어지는가 (스포일러: 거의 전부)

힘들게 얻은 당신의 Claude 습관은 낭비가 아닙니다. 이것들은 본질적으로 변함없이 이어집니다:

- **프롬프트 구조** — 명확한 역할, 명시적 작업, 예시, 요청된 출력 형식 → [프롬프팅 기초](/docs/prompting/basics)
- **컨텍스트를 의도적으로 주기** — 범위를 좁히고, 알맞은 자료를 가리키기 (이제는 실시간 Drive 문서일 수도 있음)
- **특정 출력 형식 요청하기** — 표, JSON, 단계; 똑같이 동작합니다
- **제자리에서 반복하기** — 후속 질문으로 다듬기
- **도구 사용 루프** — 도구를 기술하고, 모델이 호출하게 하고, 결과를 돌려주기 → [도구 사용](/docs/api/tool-use)
- **작업 난이도로 모델 고르기** — 빠르고 저렴하게는 Flash, 깊이 있게는 Pro, Opus/Sonnet/Haiku와 같은 판단

## 정말로 조정해야 할 것

실제로 다른 것들의 짧고 정직한 목록:

- **통합은 프롬프트가 아니라 워크플로를 바꿉니다.** Gemini에서는 붙여넣는 대신 실시간 구글 데이터를 가리킬 수 있습니다. 무엇에 접근이 허용되는지 확인하세요 — 그건 계정 및 관리자 제어 대상입니다.
- **큰 입력은 기본값을 "전부 다 집어넣기"로.** 큰 컨텍스트와 네이티브 멀티모달은 종종 청킹/검색 반사를 건너뛰고 전체 문서나 영상을 먼저 시도할 수 있게 합니다.
- **구축자에게는 두 개의 문.** Gemini 앱(어시스턴트)과 AI Studio(개발자 플레이그라운드 + 키)는 별개입니다 — 채팅 앱 안에서 API 키를 찾지 마세요.
- **도구의 와이어 포맷.** 함수 호출 *개념*은 동일하지만, JSON 세부사항은 그렇지 않습니다 — Claude의 정확한 도구 JSON을 그대로 재사용하지 마세요.
- **변동성 있는 구체 사항.** 모델 이름, 버전 번호, 등급, 컨텍스트 한도, 가격은 빠르게 바뀝니다. Claude에서와 마찬가지로 출처에서 확인하는 습관을 들이세요.

## Gemini가 빛나는 경향이 있는 때

둘 다 훌륭한 범용 어시스턴트이고 특정 작업에서의 격차는 종종 작고 계속 바뀝니다 — 그러니 이것을 법칙이 아니라 *경향*으로 다루고, 당신 자신의 [자신의 데이터에 대한 평가](/docs/models/choosing-a-model)가 결정하게 하세요:

- **구글 생태계 작업.** 업무가 이미 Gmail, 문서, Drive, 또는 캘린더에 있다면, Gemini의 네이티브 Workspace 통합은 Claude가 수동 붙여넣기 없이는 따라올 수 없는 마찰을 없애줍니다.
- **거대한 단발 컨텍스트.** *한 번에* 추론하고 싶은 엄청난 문서, 코드베이스 전체, 또는 긴 미디어 — 큰 컨텍스트의 Gemini 모델 쪽으로 기울고 검색 스캐폴딩을 건너뛰세요.
- **네이티브 멀티모달.** 영상과 오디오 이해, 또는 혼합 미디어 프롬프트는 구조적인 Gemini의 강점입니다.
- **이미 구글로 돌아가는 팀.** 협업자들이 Gems를 공유하고 Workspace에서 산다면, 그들이 있는 곳에서 만나는 것이 마찰을 이깁니다.

대부분의 경우 정직한 답: **당신(또는 당신의 팀)이 이미 세팅되어 있는 쪽.** 기술은 이동 가능합니다; 세팅 비용이 진짜 전환 비용입니다. 제공업체 중립적으로 고르는 방법은 [모델 고르기](/docs/models/choosing-a-model)와 [어떤 Claude를 써야 할까?](/docs/start-here/which-claude)를 보세요.
