Профессиональные рабочие процессы и продвинутые приёмы

Продвинутый
What you'll learn
  • Оркеструйте параллельных субагентов — и знайте, когда пора перейти к динамическому рабочему процессу, управляемому скриптом
  • Сделайте хуки детерминированным связующим звеном: линтинг при сохранении, контроль на остановке, блокировка запрещённых записей
  • Превратите ваши повторяющиеся промпты в кастомные слэш-команды, навыки и стиль вывода
  • Постройте MCP-стек и автономные конвейеры (claude -p), которые работают без присмотра в CI
  • Запускайте цикл исследовать → планировать → исполнять → проверять с дисциплиной и держите контекст лёгким
  • Относитесь к CLAUDE.md как к коду: коротко, отчищено и с высоким сигналом

Это страница, где примитивы перестают быть отдельными функциями и начинают становиться рабочим процессом. Если вы уже знаете, что такое субагент или хук, то рычаг — в том, как вы их объединяете: слэш-команда, запускающая проход в режиме планирования, субагенты, распределяющие работу веером, хук, который не даст завершить ход, пока тесты не пройдут, и автономный вызов, который выполняет всё это в CI. Давайте соединим это вместе.

Последняя проверка: 2026-06-24.

Claude Code развивается быстро: флаги, встроенные команды (/deep-research, /code-review), ключевое слово ultracode, доступность динамических рабочих процессов и уровни усилий — всё это меняется между версиями. Количество звёзд у сообществ-репозиториев меняется ежедневно. Сверяйте всё, что зависит от версии, с официальной документацией, прежде чем на это полагаться.

Сверьтесь с официальным источником.

Ментальная модель: кто держит план?​

Каждый продвинутый приём ниже — это разный ответ на один вопрос: кто держит план и промежуточные результаты? Собственная формулировка Anthropic для новой функции динамических рабочих процессов чётко это раскладывает:

ИнструментКто решает, что запускается дальшеГде живут результатыМасштаб
СубагентыClaude, ход за ходомКонтекстное окно ClaudeНесколько за ход
НавыкиClaude, следуя промптуКонтекстное окно ClaudeКак и у субагентов
Команды агентовВедущий агент, ход за ходомОбщий список задачНесколько долгоживущих равноправных
Динамические рабочие процессыСкриптПеременные скриптаОт десятков до сотен

Прогрессия — это вся суть игры: начните с одного Claude, делегируйте субагентам, чтобы защитить контекст, и переносите план в код только тогда, когда задаче нужно больше агентов, чем может скоординировать один разговор.

Приём 1 — Параллельные субагенты, затем переход к рабочему процессу​

Субагент — это отдельный Claude со своим контекстным окном и ограниченным набором инструментов; он возвращает результат, а не свою стенограмму. Привычка опытного пользователя — распределять веером независимую работу:

Распределить ревью по модулям веером

Review the changes in auth/, billing/, and api/ — use the code-reviewer
subagent on each, in parallel. Report only correctness bugs and missing tests.

Три рецензента работают одновременно, каждый тратит свой контекст на диффы, и ваша основная сессия видит три аккуратных отчёта вместо трёх сырых диффов. Подвох: параллелизм помогает только для независимых подзадач. Если шагу B нужен результат шага A, выполняйте их последовательно; если они пишут в одни и те же файлы, изолируйте их в git worktrees.

Когда задача перерастает один разговор — миграция на 500 файлов, общекодовый прогон по поиску багов, исследование, перекрёстно проверенное по множеству источников — переходите к динамическому рабочему процессу. Claude пишет JavaScript-скрипт оркестрации (распределить веером → свернуть → синтезировать), среда выполнения исполняет его в фоне с до 16 параллельными агентами, и в ваш контекст попадает только финальный ответ. Запустите его ключевым словом ultracode или просто попросите обычными словами:

Запустить динамический рабочий процесс

ultracode: audit every API endpoint under src/routes/ for missing auth checks,
then cross-check each finding with a second agent before reporting

Убойная фича — не просто больше агентов, а то, что скрипт может применить воспроизводимый паттерн качества, например когда независимые агенты состязательно проверяют находки друг друга, прежде чем хоть одна будет доложена. Попробуйте встроенную команду /deep-research <question>, чтобы увидеть этот паттерн вживую: она голосует по каждому утверждению и отсеивает те, что не выдерживают перекрёстной проверки.

Pro tip
  • Сохраните хороший прогон рабочего процесса как команду: откройте /workflows, выберите прогон, нажмите s. Он становится /ваше-имя в каждой будущей сессии.
  • Оцените стоимость сперва на срезе — одна директория, а не весь репозиторий. Прогон может породить гораздо больше агентов (и токенов), чем один разговор.

Приём 2 — Хуки: детерминированное связующее звено​

Инструкции в CLAUDE.md носят рекомендательный характер — Claude обычно им следует. Хуки детерминированы — они запускают скрипт в фиксированной точке цикла, гарантированно, каждый раз. Тянитесь к хуку, когда что-то должно произойти без единого исключения.

Три хук-паттерна с наибольшим рычагом:

Guided walkthrough1 of 3
  1. После того как Claude отредактирует файл, автоматически запускайте ваш форматтер или линтер, чтобы кодовая база никогда не дрейфовала. Обратная связь также возвращается к Claude, поэтому он сам себя корректирует.
  2. Хук Stop запускает ваш скрипт теста/сборки и блокирует завершение хода, пока он не пройдёт. Именно это позволяет автономному прогону завершиться корректно, вместо того чтобы остановиться, когда работа просто 'выглядит готовой'.
  3. Запретите записи в защищённый путь (миграции, генерируемые файлы, секреты) до того, как они произойдут, независимо от намерений Claude.

Вам не нужно вручную писать JSON. Попросите Claude написать хук за вас:

Пусть Claude напишет хук

Write a hook that runs eslint --fix after every file edit, and a second hook
that blocks any Write or Edit to the db/migrations/ folder. Add them to
.claude/settings.json and show me the config.
Watch out
  • Хук Stop, который продолжает блокировать, будет переопределён после нескольких последовательных блокировок, чтобы сессия не зашла в тупик — ваш контроль это защитное ограждение, а не бесконечный цикл.
  • Хуки запускаются с правами вашей оболочки. Просматривайте любой хук-скрипт перед его коммитом, ровно как вы делали бы с конфигом CI.

Приём 3 — Кастомные слэш-команды, навыки и стили вывода​

Всё, что вы запрашиваете дважды, должно стать примитивом. Решение простое: навыки — это знание, хуки — это гарантии, MCP — это действие, слэш-команды — это точки входа.

Слэш-команда (или навык с disable-model-invocation: true) упаковывает воспроизводимый рабочий процесс, который вы запускаете вручную. $ARGUMENTS делает его параметризованным:

.claude/skills/fix-issue/SKILL.md

---
name: fix-issue
description: Triage and fix a GitHub issue end-to-end
disable-model-invocation: true
---
Fix GitHub issue $ARGUMENTS:
1. gh issue view to read the issue
2. Search the codebase for the relevant files
3. Write a failing test that reproduces the bug
4. Implement the fix, then run tests and lint until green
5. Commit with a descriptive message and open a PR

Запустите её командой /fix-issue 1234. Используйте disable-model-invocation: true для всего, у чего есть побочные эффекты, что вы хотите запускать осознанно, а не давать Claude тянуться к этому самому.

Стили вывода меняют то, как Claude общается на протяжении всей сессии — лаконично для эксперта, подробно с объяснениями для обучения или в структурированном формате, который может разобрать ваша оснастка. Соедините кастомную слэш-команду (что) со стилем вывода (как) — и вы оформили оба конца взаимодействия.

Самый недооценённый продвинутый приём из официальной документации: дайте Claude взять у вас интервью перед крупной фичей, а затем начните свежую сессию, чтобы строить по записанной спецификации.

Спецификация через интервью, затем сборка в чистом контексте

I want to build [brief description]. Interview me in detail using the
AskUserQuestion tool — technical implementation, UI/UX, edge cases, tradeoffs.
Dig into the hard parts I might not have considered. When we've covered
everything, write a complete, self-contained spec to SPEC.md.

Приём 4 — Постройте MCP-стек​

MCP-серверы — это исполняемые процессы, которые Claude вызывает по JSON-RPC, чтобы реально что-то делать — запрашивать вашу базу данных, читать Sentry, подтягивать дизайн из Figma, заводить задачу в Linear. «Стек» опытного пользователя — это небольшой, продуманный набор, а не всё, что вы смогли найти:

Добавьте серверы в ваш стек

claude mcp add --transport stdio sentry -- npx -y @sentry/mcp-server
claude mcp add --transport http linear https://mcp.linear.app/sse

Два принципа держат MCP-стек быстрым:

  • Предпочитайте CLI там, где он есть. gh, aws, gcloud и sentry-cli — самый контекстно-эффективный способ общаться с сервисом: Claude их уже знает, и они не загружают схему инструментов в каждый ход. Резервируйте MCP для сервисов без хорошего CLI или там, где вам нужен структурированный, типизированный доступ.
  • Держите список серверов лёгким. Определения инструментов каждого подключённого сервера потребляют контекст заранее. Обрезайте серверы, которыми вы активно не пользуетесь; раздутые списки инструментов вытесняют ваши настоящие инструкции.

Для более глубокого «почему» см. Инженерия контекста — та же логика бюджета внимания, что управляет CLAUDE.md, управляет и вашим списком инструментов.

Приём 5 — Автономные конвейеры и Agent SDK​

claude -p "prompt" запускает Claude неинтерактивно — без сессии, с разбираемым выводом — и это дверь в CI, pre-commit хуки и пакетные задания. Это поверхность автономного режима / Agent SDK.

Классический паттерн распределения-веером-по-файлам из официального руководства по лучшим практикам:

Автономная пакетная миграция

# 1. Have Claude generate the work list first, then loop:
for file in $(cat files.txt); do
claude -p "Migrate $file from React to Vue. Return OK or FAIL." \
  --allowedTools "Edit,Bash(git commit *)"
done

Два флага делают всю тяжёлую работу: --output-format json (или stream-json --verbose) делает результаты машиночитаемыми, а --allowedTools точно ограничивает, к чему Claude может прикасаться, когда никто не наблюдает. Направляйте это куда угодно:

Claude как этап конвейера

cat error.log | claude -p "Cluster these errors by root cause, output JSON" \
--output-format json | jq '.[] | select(.severity=="high")'
Watch out
  • Всегда тестируйте промпт на 2–3 элементах, прежде чем спускать цикл на 2 000. Доработайте, затем масштабируйте.
  • Автономным прогонам нужна встроенная верификация — хук Stop или шаг с тестом прямо в промпте — иначе вы только что автоматизировали производство правдоподобно-но-неверного вывода в масштабе.

Приём 6 — Дисциплина режима планирования и инженерия контекста​

Самый надёжный рычаг качества — отделение исследования от исполнения. Четырёхфазный цикл из руководства Anthropic по лучшим практикам:

Guided walkthrough1 of 4
  1. Войдите в режим планирования. Claude читает файлы и отвечает на вопросы, но не вносит изменений. Направьте его на точные директории: 'read /src/auth and explain how sessions work.'
  2. Попросите детальный план реализации. Отредактируйте его прямо перед утверждением — план, который вы исправили, превосходит план, который вы пробежали глазами.
  3. Выйдите из режима планирования. Claude кодит по плану и запускает указанную вами верификацию.
  4. Пусть субагент со свежим контекстом проверит дифф относительно плана, исправит пробелы, затем закоммитит и откроет PR.

См. Режим планирования для дисциплины и пропускайте его, когда дифф умещается в одно предложение — у планирования есть накладные расходы. Причина, почему это работает, — экономия контекста: производительность деградирует по мере заполнения окна, поэтому инструкция на 50 токенов в контексте на 2 000 токенов бьёт куда сильнее, чем та же инструкция, погребённая в 50 000 токенов. Практические привычки:

  • /clear между несвязанными задачами; чистая сессия с лучшим промптом превосходит длинную загрязнённую.
  • После двух неудачных исправлений перестаньте исправлять — /clear и перепишите промпт с учётом того, что узнали.
  • Делегируйте исследование субагентам, чтобы чтение файлов происходило в их контексте, а не в вашем.

Путь «прокачай свою настройку»​

Если вы не сделаете ничего другого, сделайте это по порядку:

Guided walkthrough1 of 6
  1. Запустите /init для стартового варианта, затем вырежьте каждую строку, не прошедшую тест: 'привело бы удаление этого к тому, что Claude совершит ошибку?' Раздутый файл заставляет Claude игнорировать правила, которые важны.
  2. Добавьте хук PostToolUse для линтинга/форматирования и хук Stop, который контролирует на тестах. Теперь цикл замыкается сам, а не ждёт вас.
  3. Выберите рабочий процесс, который вы набирали дважды — fix-issue, write-tests, ship-pr — и положите его в .claude/skills/ или .claude/commands/.
  4. Добавляйте только те серверы, которыми пользуетесь еженедельно; предпочитайте CLI gh/aws вместо MCP там, где они есть. Остальное обрежьте.
  5. Используйте режим планирования для любого многофайлового или незнакомого изменения и завершайте субагентом-ревьюером со свежим контекстом.
  6. Запустите claude -p в pre-commit хуке или небольшом пакетном задании с --allowedTools, чтобы прочувствовать Claude как этап конвейера, а не просто как чат.

Мастерство CLAUDE.md​

CLAUDE.md загружается в начале каждого разговора, поэтому это самый высокочастотный контекст, который вы контролируете — и потому его легче всего испортить переполнением.

Класть в CLAUDE.mdДержать снаружи
Bash-команды, которые Claude не может угадатьВсё, что Claude находит, читая код
Правила стиля кода, отличающиеся от дефолтовСтандартные соглашения, которые Claude уже знает
Тест-раннер + как запустить один тестПолную документацию по API (вместо этого ссылка)
Этикет репозитория (именование веток/PR)Информацию, которая часто меняется
Неочевидные подводные камни и причуды окруженияБанальности «пиши чистый код»

Относитесь к нему как к коду: пересматривайте, когда поведение идёт не так, регулярно отчищайте и используйте импорты @path/to/file плюс CLAUDE.md по директориям, чтобы каждая часть монорепозитория получала только то, что релевантно. Переносите иногда релевантное знание в навыки, чтобы оно загружалось по запросу, а не облагало налогом каждый ход.

Key takeaways
  • Прогрессия и есть навык: один Claude → параллельные субагенты → управляемые скриптом динамические рабочие процессы, выбираемые по тому, кому нужно держать план.
  • Хуки — это детерминированное связующее звено — линтинг при сохранении, контроль на остановке, блокировка запрещённых записей — для того, что должно происходить каждый раз.
  • Упаковывайте повторяющиеся промпты в слэш-команды/навыки, оформляйте подачу стилем вывода и держите MCP-стек лёгким (предпочитайте CLI).
  • Автономный claude -p с --allowedTools превращает Claude в этап CI/конвейера; всегда встраивайте верификацию для автономных прогонов.
  • Дисциплина режима планирования плюс агрессивная экономия контекста (/clear, исследование субагентами, отчищенный CLAUDE.md) — самый надёжный рычаг, который у вас есть.
