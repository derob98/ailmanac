Bewerte deinen Claude-Agenten (Evals)

Experte

Du hast einen Prompt angepasst und er fühlt sich besser an — aber stimmt das? Ohne Evals (Evaluierungen) fliegst du blind: Jede Änderung ist ein Münzwurf, und du erfährst von einem verärgerten Nutzer, dass etwas kaputt ist, nicht von einem Test. Evals verwandeln "Bauchgefühl" in eine Zahl, der du vertrauen, die du verteidigen und über die Zeit beobachten kannst. Das ist der größte einzelne Unterschied zwischen Hobby-Prompts und produktionsreifer Claude-Arbeit.

What you'll learn
  • Warum "sieht für mich gut aus" kein Test ist — und was du stattdessen messen solltest
  • Baue einen Golden Dataset aus ECHTEN Fehlern (bottom-up), nicht aus erfundenen
  • Bewerte mit Code, wo du kannst, und mit einem LLM-as-Judge, wo du es nicht kannst
  • Verdrahte Evals in die CI, damit eine Prompt- oder Modelländerung niemals still regredieren kann
Zuletzt verifiziert: 2026-06-24.

Eval-Tooling und Modellverhalten bewegen sich schnell — prüfe die verlinkten Guides erneut und kalibriere deinen Judge neu, sobald du das zu testende Modell änderst. OWASP listet das Falschmachen davon (sowie Prompt Injection) weiterhin unter den größten LLM-Risiken für 2026.

Bestätige anhand der offiziellen Quelle.

Die Denkweise: messen, nicht raten​

Drei Regeln, die dich retten:

  • Bottom-up schlägt top-down. Sammle zuerst tatsächliche Fehler und entwirf dann die Metrik, um sie abzufangen. Ein Eval, das aus echten Pannen entsteht, sagt echte Pannen voraus; ein Eval, das am Whiteboard erfunden wurde, misst meist nur deine Vorstellungskraft.
  • Eine Zahl, die du erneut ausführen kannst. Ein Eval ist wiederholbar: gleiche Eingaben → vergleichbarer Score. Genau das erlaubt dir, Prompt v1 vs. v2 oder claude-haiku-4-5 vs. claude-sonnet-5 ehrlich zu vergleichen.
  • Günstig auszuführen, oft auszuführen. Wenn ein Mensch dafür einen Nachmittag braucht, passiert es nicht. Automatisiere es.

Baue einen Golden Dataset (bottom-up)​

Dein Golden Dataset ist das Herz jedes Evals — eine kuratierte Menge an Eingaben mit bekannt-guten Erwartungen.

Guided walkthrough1 of 4
  1. Beginne mit tatsächlich schlechten Ausgaben: Produktions-Traces, Bug-Reports, Support-Tickets. Das sind die Fälle, die zählen.
  2. Schreibe von Hand Fälle, die deine kritischsten und fehleranfälligsten Szenarien abdecken. Das ist dein stabiler Anker-Set.
  3. Füge anonymisierte Produktionsbeispiele (entferne PII) und synthetische Fälle für unterrepräsentierte Szenarien hinzu. Vertraue keinen Aggregat-Metriken auf einem winzigen Set.
  4. Jede neue Produktionsregression wird zu einem neuen Testfall. Ein Golden Dataset lebt, er ist nicht eingefroren.

Bewerten: zuerst Code, dann Judge​

Greife zuerst zur günstigsten zuverlässigen Prüfung.

  • Programmatische (deterministische) Prüfungen — nutze diese überall dort, wo die Antwort eine Struktur hat: exakter/Keyword-Abgleich, "gültiges JSON gegen dieses Schema", "hat es das richtige Tool mit den richtigen Argumenten aufgerufen", "unter N Tokens / unter X ms". Schnell, kostenlos und nie unzuverlässig.
  • LLM-as-Judge — für unscharfe Dimensionen (Hilfreichkeit, Tonfall, Treue zu einer Quelle), die sich Code widersetzen. Gib dem Judge eine Rubrik, kein Bauchgefühl, und kalibriere ihn gegen menschliche Labels, bevor du ihm vertraust.

:::warning Judges haben Verzerrungen LLM-Judges driften zu längeren Antworten (Verbosity Bias) und zu der Option, die zuerst gezeigt wird (Position Bias). Gegenmaßnahmen: eine strenge Rubrik, paarweiser Vergleich statt absoluter Bewertung, Vertauschen der Antwortreihenfolge und erneutes Prüfen des Judges gegen einen menschlich gelabelten Ausschnitt. Ein Judge ist eine Schicht, nicht der gesamte Test. :::

LLM-as-Judge-Rubrik (Starter)

You are a strict grader. You are given a QUESTION, a REFERENCE answer, and a MODEL answer.
Score the MODEL answer from 1-5 on (a) faithfulness to the reference and (b) helpfulness.
Output ONLY JSON, nothing else: {"score": <1-5>, "reason": "<one short sentence>"}

QUESTION: {{question}}
REFERENCE: {{reference}}
MODEL: {{model_answer}}

Bei Agenten teste auch die Trajektorie​

Ein Agent kann die richtige Endantwort auf dem falschen Weg erreichen — in Schleifen laufen, ein destruktives Tool aufrufen oder dein Budget verbrennen. Bewerte also den Weg, nicht nur das Ziel: Hat es die richtigen Tools in einer sinnvollen Reihenfolge aufgerufen, ohne Schleifen, innerhalb des Budgets? Tool-Call-Korrektheit und Trajektorien-Prüfungen fangen Fehler ab, die ein Eval, das nur die Endantwort betrachtet, niemals sieht.

Verdrahte es in die CI​

Hier zahlen sich Evals aus: Mache Regressionen unmöglich zu mergen.

Guided walkthrough1 of 3
  1. Bewerte programmatisch, wo möglich; lass den Judge auf den Rest los.
  2. Setze eine Schwelle (z. B. der Score darf nicht gegenüber main sinken). Eine Prompt-Änderung, die die Qualität verschlechtert, kann nicht ausgeliefert werden.
  3. Wenn ein Judge eine Live-Antwort markiert, leite sie in eine menschliche Review-Queue; der Reviewer bestätigt, fügt den Fall dem Golden Set hinzu und testet nach dem Fix erneut.

Key takeaways
  • Kein Eval = Ausliefern nach Bauchgefühl. Baue eines, bevor du einem Prompt oder Agenten vertraust.
  • Golden Dataset aus echten Fehlern; lass ihn jede Woche aus neuen Regressionen wachsen.
  • Code-basierte Prüfungen zuerst; LLM-as-Judge (mit Rubrik, kalibriert) für die unscharfen Teile.
  • Bei Agenten bewerte die Trajektorie, nicht nur die Ausgabe.
  • Führe es in der CI aus und lass den Build bei einem Abfall fehlschlagen — so hört Qualität auf zu regredieren.
