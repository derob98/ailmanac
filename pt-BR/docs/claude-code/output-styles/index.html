Estilos de Saída

Intermediário
Última verificação: 2026-06-21.

Os estilos integrados, o mecanismo exato de troca e o formato de arquivo de estilo personalizado podem mudar entre versões. Confirme na documentação oficial (a documentação do Claude Code foi movida de docs.anthropic.com para code.claude.com).

Confirme na fonte oficial.
What you'll learn
  • O que é um estilo de saída — e como ele difere de mudar o que o Claude sabe
  • Os quatro estilos integrados: Default, Proactive, Explanatory, Learning
  • Como trocar de estilo com /config — e por que a mudança precisa de uma nova sessão
  • Como criar seu próprio estilo personalizado como um arquivo Markdown

Os estilos de saída mudam como o Claude Code se comunica — seu papel, tom, verbosidade e se ele explica seu raciocínio — modificando o prompt do sistema diretamente. Eles mudam como o Claude responde, não o que ele sabe. Recorra a um quando você ficar reescrevendo o prompt pedindo a mesma voz ou formato a cada turno.

Por que usá-los​

  • Adeque-se ao seu fluxo de trabalho. Conciso e orientado à ação quando você está avançando rápido; explicativo quando está aprendendo.
  • Modo de ensino. Alguns estilos adicionam comentários educativos — ótimos enquanto você está conhecendo um codebase ou uma técnica.
  • Consistência. Defina um estilo preferido uma vez para que cada sessão seja lida do jeito que você gosta.

Os estilos integrados​

O Claude Code vem com quatro:

EstiloO que faz
DefaultO prompt de sistema padrão de engenharia de software — eficiente, orientado à ação.
ProactiveExecuta imediatamente e faz suposições razoáveis em vez de pausar para decisões rotineiras. Orientação de execução autônoma mais forte do que o modo automático, mas você ainda vê solicitações de permissão antes que as ferramentas sejam executadas.
ExplanatoryAdiciona "Insights" educativos entre as etapas, explicando escolhas de implementação e padrões do codebase.
LearningColaborativo, aprender-fazendo: compartilha Insights e pede que você escreva pequenos trechos por conta própria, deixando marcadores [TODO(human)] no seu código.
Watch out
  • Explanatory e Learning produzem respostas mais longas por design, então custam mais tokens de saída do que o Default.

Como trocar​

Execute /config e escolha um estilo em Output style. Sua escolha é salva em .claude/settings.local.json no nível do projeto. Para defini-lo sem o menu, edite o campo outputStyle em qualquer arquivo de configurações.

Guided walkthrough1 of 4
  1. Execute /config na sua sessão e encontre a opção Output style.
  2. Selecione um entre Default, Proactive, Explanatory ou Learning. Sua escolha é salva em .claude/settings.local.json no nível do projeto.
  3. Pule o menu editando o campo outputStyle em qualquer arquivo de configurações (veja o JSON abaixo).
  4. O estilo faz parte do prompt do sistema, lido uma única vez no início da sessão — execute /clear ou abra uma nova sessão para que ele entre em vigor.
{
  "outputStyle": "Explanatory"
}

:::warning As mudanças exigem uma nova sessão O estilo de saída faz parte do prompt do sistema, que o Claude Code lê uma vez no início da sessão. Uma mudança só tem efeito após /clear ou uma nova sessão. (O comando autônomo /output-style foi removido na v2.1.91 — use /config ou a configuração acima.) :::

Crie um estilo personalizado​

Um estilo personalizado é um arquivo Markdown: frontmatter e, em seguida, as instruções a serem anexadas ao prompt do sistema. Salve-o no nível do usuário (~/.claude/output-styles/) ou no nível do projeto (.claude/output-styles/); o nome do arquivo se torna o nome do estilo, a menos que você defina name.

Exemplo de estilo personalizado — Diagramas primeiro

---
name: Diagrams first
description: Lead every explanation with a diagram
keep-coding-instructions: true
---

When explaining code, architecture, or data flow, start with a Mermaid
diagram showing the structure, then explain in prose.
Pro tip
  • Defina keep-coding-instructions: true quando você estiver mudando apenas COMO o Claude se comunica, mas ainda quiser que ele programe normalmente.
  • Deixe-o de fora (o padrão) quando o Claude não estiver fazendo engenharia de software de forma alguma — digamos, um assistente puramente de escrita ou de análise de dados — para que as instruções de programação integradas sejam descartadas.

Estilos de saída vs CLAUDE.md vs outras alavancas​

  • Estilo de saída = uma predefinição de prompt do sistema para papel, tom e formato — aplica-se a cada resposta.
  • CLAUDE.md = fatos e regras do projeto persistentes (convenções, salvaguardas), adicionados como uma mensagem após o prompt do sistema.
  • Instruções personalizadas / estilos de resposta = o equivalente no app de consumo Claude.ai.

:::tip Use a alavanca certa Quer que o Claude "sempre execute os testes antes de finalizar"? Isso é uma salvaguarda do CLAUDE.md, não um estilo. Quer que ele "explique cada etapa conforme avança"? Isso é um estilo de saída. Mantenha as regras de comportamento no CLAUDE.md e a apresentação nos estilos. :::

Key takeaways
  • Os estilos de saída mudam COMO o Claude se comunica (papel, tom, verbosidade), não O QUE ele sabe — eles editam o prompt do sistema.
  • Quatro integrados: Default (orientado à ação), Proactive (executa com suposições), Explanatory (adiciona Insights), Learning (Insights + marcadores [TODO(human)]).
  • Explanatory e Learning custam mais tokens de saída porque as respostas são mais longas por design.
  • Troque via /config (salvo em .claude/settings.local.json) ou a configuração outputStyle — as mudanças precisam de /clear ou uma nova sessão.
  • Estilos personalizados são arquivos Markdown em ~/.claude/output-styles/ ou .claude/output-styles/; use keep-coding-instructions: true para manter o comportamento de programação.
  • Use CLAUDE.md para regras de comportamento/salvaguardas e estilos de saída para apresentação.
