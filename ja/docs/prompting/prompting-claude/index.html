Claude 固有のプロンプティング

中級
最終検証: 2026-06-20。

Anthropic のプロンプトエンジニアリングのガイダンスはモデルごとに進化します — 現在のベストプラクティスは公式のプロンプトエンジニアリングのドキュメントを確認してください。

次の情報源で確認してください: 公式情報源.

プロンプティングのほとんどは普遍的です。しかし、いくつかの手法は特に Claude で効果的であり — そしていくつかの古いモデルの習慣は今や逆効果です。このレッスンは、ノイズから信号を分けます: 何が本当に Claude 固有なのか。

What you'll learn
  • 活かす価値のある Claude 固有の手法を見分ける — XML タグ、明確さ、役割、思考、例、プリフィル
  • 今や裏目に出る古いモデルの習慣（ALL-CAPS、脅し、過剰に作り込んだ足場）を捨てる
  • Claude の特徴として持ち歩くべき 2 つの習慣 — XML タグへの親和性とプリフィル — を知る
  • モデルを切り替えたときに何を再確認すべきかを理解する

Claude ではこれらを活かす​

これらは Claude で最も効果のある動きです。最初の 1 つは、身につけられる最も価値の高い習慣です。

  • XML タグ。 Claude は、入力・指示・例に名前付きタグを使った構造化プロンプトによく調整されています。これは最も価値の高い Claude の習慣です。
  • 明確かつ率直に。 目標、制約、そして「完了」がどう見えるかを述べる。Claude は明示的な指示に忠実に従います。
  • 役割を与える — システムプロンプトを通じて。
  • 難しい問題では考えさせる — ステップごとの推論を求めるか、推論モデルでは拡張思考 / エフォートを使う（その場合、明示的に CoT を求める必要がないことがよくあります）。
  • 例を使う（フューショット）— 形式やエッジケースのために。
  • プリフィル — Claude の応答の冒頭（API 上）を先に書いて形式を誘導する。

プリフィルを試す​

プリフィルとは、アシスタントのターンを自分で始めて、欲しい形式で Claude に続けさせることです。API 上では、開き文字や見出しでアシスタントのターンを始めます:

これらの古い習慣を捨てる​

新しく、字義どおりに従う Claude モデルはルールを変えます。古いモデルで役立ったトリックは今や邪魔になります。

Watch out
  • ALL-CAPS や脅し口調のプロンプト（「YOU MUST… OR ELSE」）を控える。新しい Claude モデルは指示に字義どおり従い、丁寧な言い回しの具体的な指示の方が、怒鳴るよりうまくいく。
  • 手の込んだ「ジェイルブレイク風」の足場で過剰に作り込まない — 明確で正直な指示がトリックに勝る。
  • 少ない方がよいことが多い — 引き締まったプロンプトは、頼みごとを埋もれさせる冗長なプロンプトに勝る。

Claude 対他のモデル​

原則は転用できます（クロス AI プロンプト翻訳を参照）が、Claude 固有として最も覚えておく価値のあるものが 2 つあります。

Guided walkthrough1 of 3
  1. 入力、指示、例を名前付きタグで囲む。Claude のタグ親和性により、これが最も信頼できる構造化の習慣になる。
  2. アシスタントのターンを仕込んで形式を固定する — JSON には開き波括弧、構造には見出し。
  3. Claude から離れるときは、口調の強さと書式の手がかりを再確認する。ここで字義どおりなものが、きれいに転用されるとは限らない。

Key takeaways
  • XML タグは最も価値の高い Claude の習慣。プリフィルは最も役立つ Claude 固有の形式レバー。
  • 新しい Claude モデルは指示に字義どおり従う — 大声や脅しではなく、明確かつ率直に。
  • 少ない方がよい: 引き締まった正直なプロンプトは、手の込んだ足場に勝る。
  • 原則はモデル間で転用できるが、切り替えたら口調の強さと書式の手がかりを再確認する。
