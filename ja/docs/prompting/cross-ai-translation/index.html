AI横断のプロンプト翻訳

中級
What you'll learn
  • どのプロンプティングの基本原則が、あらゆるモデルにそのまま引き継げるかを知る
  • モデルやベンダーを切り替えたときに違ってくる、ごく少数の表面的な細部を見抜く
  • 書き直さずにプロンプトを新しいモデルへ移せる、持ち運べるワークフローを実行する
  • 推論モデルに『ステップごとに考えて』と二重にプロンプトを与える罠を避ける

良い知らせがあります。プロンプティングの基本原則は、どのモデルにも引き継げます。 Claudeを上手にプロンプトできるなら、ChatGPT、Gemini、Llamaも上手にプロンプトできます。表面的な細部はいくつか異なります — これがそのロゼッタストーンです。

どこでも引き継げるもの​

これらはどのモデルでも機能します — ベンダーではなくコミュニケーションに関わるものだからです。

  • 背景を与える: 目的、読み手、形式、トーン（プロンプティングの基礎）。
  • 例（few-shot）。
  • 難しいタスクでは推論を求める。
  • 出力を制約する（長さ、形式、「JSONのみ」）。
  • 平易な言葉で反復する。
  • 役割 / システムプロンプト — 主要なモデルにはすべてこの概念があります（役割）。

一度これらを身につければ、どこにでも持ち運べます。

異なるもの（と調整すべきこと）​

観点切り替え時の注意点
構造の手がかりClaudeはXMLタグを好みます。他のモデルも受け付けますが、Markdownの見出しやJSONを好む場合があります。
トーンの強さ古いモデルは強い口調やALL-CAPSのプロンプトを許容しました。指示に忠実な新しいモデルは、落ち着いた具体的な指示を好みます。モデルごとに調整し直しましょう。
推論「ステップごとに考えて」は一部のモデルで役立ちます。専用の推論モデルは内部で行うので — 重ねて指示しないでください。
プレフィルアシスタントの発言の冒頭に言葉を入れるのはClaude流のレバーです。他では対応状況がまちまちです。
システムプロンプトの重みすべてに存在しますが、どれだけ強く従われるかは異なります — テストしましょう。
ツール / 関数呼び出し概念は同じですが、リクエストのスキーマはプロバイダーごとに異なります。
6つの表面的な違い
Enter キーまたはスペースキーでカードを裏返します。左右の矢印キーでカードを移動できます。用語を表示しました。

持ち運べるワークフロー​

Guided walkthrough1 of 4
  1. 背景 + 形式 + 例で、つまりどのモデルにも引き継げる部分でプロンプトを起草します。まだベンダー固有の構文を組み込まないでください。
  2. 実際に投入するモデルで、いくつかの実際の入力を通します（/docs/foundations/evals）。モデルに対する自分の直感ではなく、出力を評価しましょう。
  3. 何かがおかしければ、プロンプト全体ではなく、表面のレバーを一度に一つずつ変えます — 区切り記号のスタイル、トーンの強さ、推論を求めるかどうか。
  4. プロンプトをファイルとして保存すれば、モデルの切り替えは書き直しではなく、小さくレビュー可能な差分になります。

表面だけが変わるので、うまく構造化された一つのプロンプトはたいてい一行の微調整で移植できます。以下は普遍的なやり方で書かれた一例です — 背景、明示的な形式、明確なタスクで、剥ぎ取るべきベンダー固有の構文はありません。

持ち運べる、ベンダー中立なプロンプト

You are a technical editor. Rewrite the text below to be clearer and more concise.

Constraints:
- Keep every fact unchanged.
- Target a general professional audience.
- Return only the rewritten text, no preamble.

Text:
"""
[paste the draft here]
"""

そのプロンプトはClaude、GPT、Gemini、あるいはオープンモデルでもそのまま動きます。もしあるモデルが「書き直した文章だけを返す」を無視するなら、それがあなたの唯一の表面的な微調整です — 制約を締めるか、Claudeなら返信をプレフィルしましょう。

:::tip ゼロから書き直さないこと 新しいモデルへの移行で、新しいプロンプトが必要になることはまれです — たいていはトーンや形式の微調整だけです。構造と例はそのまま引き継がれます。 :::

理解度チェック

0/3
  1. 1次のうち、モデル間で引き継げず、切り替えたときに調整し直す必要があるものはどれですか？
  2. 2プロンプトを専用の推論モデルに移します。『ステップごとに考えて』という指示はどうすべきですか？
  3. 3プロンプトを新しいモデルに切り替える推奨される方法は何ですか？
Key takeaways
  • 基本原則 — 背景、例、推論、出力の制約、役割 — はすべてのモデルにそのまま引き継げる。
  • 違うのは6つの表面的な細部だけ: 構造の手がかり、トーンの強さ、推論プロンプト、プレフィル、システムプロンプトの重み、ツールのスキーマ。
  • モデルの切り替えは書き直しではなく微調整: 普遍的な構造を保ち、対象でテストし、表面のレバーを一度に一つずつ調整する。
  • プロンプトをバージョン管理して、モデル間の移動を小さくレビュー可能な差分にする。
