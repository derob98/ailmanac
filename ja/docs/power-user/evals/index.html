Claude エージェントを評価する（Evals）

🔴上級

プロンプトを少しいじったらなんとなく良くなった気がする — でも本当に？ Eval（評価）がなければ手探り状態です。あらゆる変更がコイン投げになり、壊れたことは怒ったユーザーからではなく、テストからではなく知ることになります。Eval は「なんとなく」を、信頼でき、根拠を示せ、時間とともに見守れる数値に変えます。これこそが、趣味のプロンプトとプロダクション品質の Claude の仕事を分ける最大の違いです。

🎯What you'll learn
  • ✅「自分には良さそうに見える」がなぜテストではないのか — そして代わりに何を測るべきか
  • ✅想像上ではなく、REAL な失敗からゴールデンデータセットを構築する（ボトムアップ）
  • ✅可能なところはコードで採点し、できないところは LLM-as-judge で採点する
  • ✅プロンプトやモデルの変更がこっそりリグレッションを起こさないよう、Eval を CI に組み込む
✅
最終検証: 2026-06-24。

Eval のツールやモデルの挙動は速く動きます — リンク先のガイドを再確認し、テスト対象のモデルを変えるたびにジャッジを再キャリブレーションしてください。OWASP は今でも、これを誤ること（およびプロンプトインジェクション）を 2026 年における LLM の最大級のリスクに数えています。

次の情報源で確認してください: 公式情報源.

マインドセット：推測せず、測る​

あなたを救う 3 つのルール：

  • ボトムアップはトップダウンに勝る。 まず実際の失敗を集め、それを捕まえるよう指標を設計する。実際の故障から作った Eval は実際の故障を予測します。ホワイトボードで考案した Eval は、たいていあなたの想像力を測っているだけです。
  • 再実行できる数値。 Eval は再現可能です。同じ入力 → 比較可能なスコア。これがあるからこそ、プロンプト v1 と v2、あるいは claude-haiku-4-5 と claude-sonnet-5 を誠実に比較できます。
  • 安く実行でき、頻繁に実行する。 人間が午後いっぱいかかるなら、実行されません。自動化しましょう。

ゴールデンデータセットを構築する（ボトムアップ）​

ゴールデンデータセットはあらゆる Eval の心臓部です — 既知の正解期待値が付いた、厳選された入力のセットです。

Guided walkthrough1 of 4
  1. 実際の悪い出力から始める：プロダクションのトレース、バグレポート、サポートチケット。これらが重要なケースです。
  2. 最も重要で、最も誤りが起きやすいシナリオをカバーするケースを手で書く。これがあなたの安定したアンカーセットです。
  3. 匿名化したプロダクションサンプル（PII を除去）と、表現が不足しているシナリオ向けの合成ケースを追加する。小さなセットでの集計指標を信じてはいけません。
  4. 新しいプロダクションのリグレッションはすべて新しいテストケースになる。ゴールデンデータセットは凍結されたものではなく、生きています。

採点：コードを先に、ジャッジを後に​

まずは最も安く信頼できるチェックに手を伸ばしましょう。

  • プログラム的（決定論的）チェック — 答えに構造があるところならどこでも使う：完全一致／キーワード一致、「このスキーマに対して有効な JSON か」、「正しい引数で正しいツールを呼んだか」、「N トークン未満／X ミリ秒未満か」。高速で、無料で、決してフレーキーになりません。
  • LLM-as-judge — コードでは扱いにくい曖昧な次元（有用性、トーン、ソースへの忠実性）のために。ジャッジには雰囲気ではなくルーブリックを与え、信頼する前に人間のラベルに対してキャリブレーションしてください。

:::warning ジャッジにはバイアスがある LLM のジャッジは、より長い回答（冗長性バイアス）や、最初に表示された選択肢（位置バイアス）へと偏ります。防御策：厳密なルーブリック、絶対スコアリングではなくペアワイズ比較、回答の順序の入れ替え、そして人間がラベル付けしたスライスに対するジャッジの再チェック。ジャッジは一つの層であって、テスト全体ではありません。 :::

✦

LLM-as-judge ルーブリック（スターター）

You are a strict grader. You are given a QUESTION, a REFERENCE answer, and a MODEL answer.
Score the MODEL answer from 1-5 on (a) faithfulness to the reference and (b) helpfulness.
Output ONLY JSON, nothing else: {"score": <1-5>, "reason": "<one short sentence>"}

QUESTION: {{question}}
REFERENCE: {{reference}}
MODEL: {{model_answer}}

エージェントの場合は、*軌跡（トラジェクトリ）*もテストする​

エージェントは、正しい最終回答に間違ったやり方でたどり着くことがあります — ループしたり、破壊的なツールを呼んだり、予算を使い果たしたり。だから目的地だけでなく経路を評価しましょう：正しいツールを、まともな順序で、ループなしに、予算内で呼んだか？ ツール呼び出しの正しさと軌跡のチェックは、最終回答だけを見る Eval では決して見えない失敗を捕まえます。

CI に組み込む​

ここで Eval は報われます：リグレッションをマージ不可能にするのです。

Guided walkthrough1 of 3
  1. 可能なところはプログラム的に採点し、残りはジャッジを実行する。
  2. 閾値を設定する（例：スコアが main に対して下がってはならない）。品質をリグレッションさせるプロンプト変更は出荷できません。
  3. ジャッジが本番の応答にフラグを立てたら、人間のレビューキューに回す。レビュアーが確認し、そのケースをゴールデンセットに追加し、修正後に再テストする。

💡Key takeaways
  • ⭐Eval がない＝なんとなくで出荷している。プロンプトやエージェントを信頼する前に一つ作ること。
  • ⭐ゴールデンデータセットは実際の失敗から。新しいリグレッションで毎週育てる。
  • ⭐コードベースのチェックを先に。曖昧な部分には LLM-as-judge（ルーブリック付き、キャリブレーション済み）。
  • ⭐エージェントには、出力だけでなく軌跡を採点する。
  • ⭐CI で実行し、低下したらビルドを失敗させる — それが品質のリグレッションを止める方法です。
